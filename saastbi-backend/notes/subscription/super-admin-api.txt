SCHEMA CROSS-REFERENCE — DATA MODELS
================================================================================

SuperAdmin (prisma model: SuperAdmin):
  - id: String (cuid, primary key)
  - name: String
  - email: String (unique)
  - password: String (hashed with argon2)
  - role: SuperAdminRoles — "ADMIN" | "MARKETING" | "MANAGER" | "RELATIONSHIP_MANAGER"
  - createdAt: DateTime
  - updatedAt: DateTime

Module (prisma model: Module):
  - id: String (cuid)
  - moduleName: String (unique)
  - moduleKey: String (unique)
  - moduleDescription: String
  - isActive: Boolean (default: false)
  - createdAt: DateTime
  - updatedAt: DateTime
  - permissions: Permission[]
  - planModules: PlanModule[]

Permission (prisma model: Permission):
  - id: String (cuid)
  - action: String (e.g., "C", "R", "U", "D", "F")
  - moduleId: String (FK -> Module)
  - createdAt: DateTime
  - updatedAt: DateTime

Plan (prisma model: Plan):
  - id: String (cuid)
  - name: String
  - price: Float (INR)
  - type: PlanType — "MONTHLY" | "YEARLY" | "WEEKLY"
  - maxUsers: Int (default: 5)
  - maxStartups: Int (default: 1)
  - storageLimit: Int | null
  - features: String[]
  - razorpayPlanId: String | null
  - createdAt: DateTime
  - updatedAt: DateTime
  - planModules: PlanModule[]
  - subscriptions: Subscription[]
  - tenants: Tenant[]

PlanModule (prisma model: PlanModule):
  - id: String (cuid)
  - planId: String (FK -> Plan)
  - moduleId: String (FK -> Module)
  - createdAt: DateTime
  - updatedAt: DateTime
  - @@unique([planId, moduleId])

Tenant (prisma model: Tenant):
  - id: String (cuid)
  - organizationName: String
  - tenantKey: String (unique)
  - domain: String | null
  - status: TenantStatus — "ACTIVE" | "INACTIVE" | "SUSPENDED"
  - planId: String | null (FK -> Plan, current active plan)
  - tenantLogo: String | null
  - pageId: String | null (unique, FK -> Page)
  - createdAt: DateTime
  - updatedAt: DateTime
  - subscriptions: Subscription[]
  - invoices: Invoice[]
  - billingHistory: BillingHistory[]
  - userMemberships: IncubationUserTenant[]
  - addonServiceRequests: AddonServiceRequest[]
  - (many other relations omitted for brevity)

Subscription (prisma model: Subscription):
  - id: String (cuid)
  - tenantId: String (FK -> Tenant)
  - planId: String (FK -> Plan)
  - razorpaySubscriptionId: String (unique)
  - razorpayPaymentId: String | null
  - razorpaySignature: String | null
  - status: SubscriptionStatus — "PENDING" | "ACTIVE" | "COMPLETED" | "CANCELED" | "EXPIRED"
  - startDate: DateTime | null
  - endDate: DateTime | null (also serves as next billing date)
  - createdAt: DateTime
  - updatedAt: DateTime

Invoice (prisma model: Invoice):
  - id: String (cuid)
  - tenantId: String (FK -> Tenant)
  - invoiceNumber: String (unique, format: "INV-YEAR-HEXCODE")
  - amount: Float
  - issuedDate: DateTime (default: now())
  - dueDate: DateTime | null
  - paidDate: DateTime | null
  - status: InvoiceStatus — "UNPAID" | "PAID" | "OVERDUE"
  - pdfUrl: String | null
  - createdAt: DateTime
  - updatedAt: DateTime
  - billingHistory: BillingHistory[]

BillingHistory (prisma model: BillingHistory):
  - id: String (cuid)
  - tenantId: String (FK -> Tenant)
  - amount: Float
  - description: String | null
  - billingDate: DateTime (default: now())
  - paymentMethod: String | null ("Razorpay" | "SYSTEM_CREDIT")
  - status: BillingStatus — "SUCCESS" | "FAILED" | "PENDING"
  - invoiceId: String | null (FK -> Invoice)

Addon (prisma model: Addon):

A1. SIGN UP (Create Super Admin)
--------------------------------------------------------------------------------

Method: POST
URL: {{BASE_URL}}/api/super-admin/sign-up
Auth: None

Purpose: Register a new super admin account.

Request Body:
{
  "email": "admin@example.com",
  "name": "John Doe",
  "role": "ADMIN",
  "password": "SecurePassword123"
}

Field Validation:
- email: Required, must be valid email format
- name: Required, string
- role: Required — "ADMIN" | "MARKETING" | "MANAGER" | "RELATIONSHIP_MANAGER"
- password: Required, string (hashed with argon2 before storage)

Response (201):
{
  "success": true,
  "status": 201,
  "data": {
    "id": "cuid_admin_1",
    "name": "John Doe",
    "email": "admin@example.com",
    "role": "ADMIN"
  },
  "message": "Super admin created successfully"
}

Error (400 — user exists):
{
  "success": false,
  "status": 400,
  "data": null,
  "message": "User already exist"
}

Error (400 — invalid email):
{
  "success": false,
  "status": 400,
  "data": null,
  "message": "Email not valid."
}

Error (400 — missing fields):
{
  "success": false,
  "status": 400,
  "data": null,
  "message": " Fields are required."
}

Frontend Notes:
- Password is NOT returned in the response
- After signup, redirect to login page
- role determines dashboard access level

--------------------------------------------------------------------------------
A2. LOGIN
--------------------------------------------------------------------------------

Method: POST
URL: {{BASE_URL}}/api/super-admin/login
Auth: None

Purpose: Authenticate super admin and receive JWT tokens.

Request Body:
{
  "email": "admin@example.com",
  "password": "SecurePassword123"
}

Response (200):
{
  "success": true,
  "status": 200,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs...",
    "user": {
      "id": "cuid_admin_1",
      "name": "John Doe",
      "email": "admin@example.com",
      "role": "ADMIN"
    }
  },
  "message": "Login successful"
}

Error (401):
{
  "success": false,
  "status": 401,
  "data": null,
  "message": "Invalid email or password"
}

Frontend Notes:
- Store accessToken in memory (short-lived, ~15 min)
- Store refreshToken securely (used for refresh flow)
- Send accessToken as: Authorization: Bearer <accessToken>
- user.role determines which dashboard sections to show

A5. REFRESH TOKEN
--------------------------------------------------------------------------------

Method: GET
URL: {{BASE_URL}}/api/super-admin/refresh
Auth: Cookie (refreshToken in httpOnly cookie)

Purpose: Get a new access token using the refresh token stored in cookies.

Response (200):
{
  "success": true,
  "status": 200,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...(new)",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs...(new)",
    "user": {
      "id": "cuid_admin_1",
      "name": "John Doe",
      "email": "admin@example.com",
      "role": "ADMIN"
    }
  }
}

Error (401):
{
  "success": false,
  "status": 401,
  "data": null,
  "message": "Invalid or expired refresh token"
}

Frontend Notes:
- Call this when accessToken expires (401 response from any API)
- Server also sets new httpOnly cookies:
    accessToken cookie: maxAge 15 minutes
    refreshToken cookie: maxAge 7 days
- Both returned in body AND set as cookies
- If refresh fails, redirect to login

--------------------------------------------------------------------------------
B1. CREATE MODULE
--------------------------------------------------------------------------------

Method: POST
URL: {{BASE_URL}}/api/super-admin/create-module
Auth: Bearer Token Required

Purpose: Create a new feature module with permissions. Modules are the
building blocks that get assigned to plans.

Request Body:
{
  "moduleName": "Analytics Dashboard",
  "moduleKey": "analytics",
  "moduleDescription": "Advanced analytics and reporting tools",
  "permissions": ["C", "R", "U", "D", "F"]
}

Field Validation:
- moduleName: Required, string (must be unique across modules)
- moduleKey: Required, string (unique identifier, e.g., "analytics", "core")
- moduleDescription: Required, string
- permissions: Optional, string array
    - If omitted or empty, defaults to: ["C", "R", "U", "D", "F"]
    - C = Create, R = Read, U = Update, D = Delete, F = Full access

Response (201):
{
  "success": true,
  "status": 201,
  "data": {
    "id": "cuid_mod_1",
    "moduleName": "Analytics Dashboard",
    "moduleKey": "analytics",
    "moduleDescription": "Advanced analytics and reporting tools",
    "isActive": false,
    "createdAt": "2026-03-29T00:00:00Z",
    "updatedAt": "2026-03-29T00:00:00Z",
    "permissions": [
      { "id": "cuid_perm_1", "action": "C", "moduleId": "cuid_mod_1" },
      { "id": "cuid_perm_2", "action": "R", "moduleId": "cuid_mod_1" },
      { "id": "cuid_perm_3", "action": "U", "moduleId": "cuid_mod_1" },
      { "id": "cuid_perm_4", "action": "D", "moduleId": "cuid_mod_1" },
      { "id": "cuid_perm_5", "action": "F", "moduleId": "cuid_mod_1" }
    ]
  },
  "message": "5"
}

NOTE: The message field contains the createdPermissionsCount as a string
(this is a known quirk — the count of created permissions is passed as
the message parameter).

Error (400 — duplicate key):
{
  "success": false,
  "status": 400,
  "data": null,
  "message": "Module already exist"
}

Frontend Notes:
- Newly created modules have isActive: false by default
- Must activate via /update-module-status before they appear in dropdowns
- moduleKey must be unique — use slugified name (e.g., "analytics-dashboard")

--------------------------------------------------------------------------------
C1. CREATE PLAN
--------------------------------------------------------------------------------

Method: POST
URL: {{BASE_URL}}/api/super-admin/create-plan
Auth: Bearer Token Required

Purpose: Create a new subscription plan with modules, pricing, and limits.
Also creates the corresponding plan on Razorpay for recurring billing.

Request Body:
{
  "name": "Pro",
  "price": 999,
  "type": "MONTHLY",
  "moduleIds": ["cuid_mod_1", "cuid_mod_2"],
  "maxUsers": 20,
  "maxStartups": 5,
  "storageLimit": 10240,
  "features": ["20 Users", "5 Startups", "Priority Support"]
}

Field Validation:
- name: Required, string
- price: Required, number (in INR)
- type: Optional, string — "MONTHLY" | "YEARLY" | "WEEKLY" (default: "MONTHLY")
- moduleIds: Optional, string array of module IDs (default: [])
- maxUsers: Optional, number (default: 5)
- maxStartups: Optional, number (default: 1)
- storageLimit: Optional, number in MB (default: 1024)
- features: Optional, string array (default: [])

Response (201):
{
  "success": true,
  "status": 201,
  "data": {
    "id": "cuid_plan_2",
    "name": "Pro",
    "price": 999,
    "type": "MONTHLY",
    "maxUsers": 20,
    "maxStartups": 5,
    "storageLimit": 10240,
    "features": ["20 Users", "5 Startups", "Priority Support"],
    "razorpayPlanId": "plan_rzp_abc123",
    "createdAt": "2026-03-29T00:00:00Z",
    "updatedAt": "2026-03-29T00:00:00Z",
    "planModules": [
      {
        "id": "cuid_pm_1",
        "planId": "cuid_plan_2",
        "moduleId": "cuid_mod_1",
        "module": {
          "id": "cuid_mod_1",
          "moduleName": "Core",
          "moduleKey": "core",
          "moduleDescription": "Core platform features",
          "isActive": true
        }
      },
      {
        "id": "cuid_pm_2",
        "planId": "cuid_plan_2",
        "moduleId": "cuid_mod_2",
        "module": {
          "id": "cuid_mod_2",
          "moduleName": "Analytics",
          "moduleKey": "analytics",
          "moduleDescription": "Analytics dashboard",
          "isActive": true
        }
      }
    ]
  },
  "message": "Plan created successfully with modules and limits"
}

Frontend Notes:
- razorpayPlanId is auto-generated — do not send from frontend
- Get moduleIds from the /module-dropdown endpoint
- features array is displayed on the pricing page as bullet points
- price is always in INR (Indian Rupees)
- Razorpay plan is created in the background — if Razorpay fails,
  the entire plan creation fails

COMPLETE FRONTEND FLOWS
================================================================================

Flow 1: Authentication
  1. POST /api/super-admin/login → get tokens
  2. Store accessToken in memory
  3. GET /api/super-admin/profile → verify and get user info
  4. On 401 → GET /api/super-admin/refresh → retry
  5. GET /api/super-admin/logout → clear tokens

Flow 2: Module Management
  1. GET /api/super-admin/get-modules → list all modules
  2. POST /api/super-admin/create-module → create new module
  3. PATCH /api/super-admin/update-module-status → toggle active/inactive
  4. PATCH /api/super-admin/edit-module → update name/description
  5. GET /api/super-admin/module-dropdown → for plan forms

Flow 3: Plan Management
  1. GET /api/super-admin/module-dropdown → get modules for form
  2. POST /api/super-admin/create-plan → create new plan
  3. GET /api/super-admin/get-plans → view all plans
  4. GET /api/super-admin/get-plans-by-type → for pricing page
  5. PUT /api/super-admin/update-plan-modules → change plan modules
  6. PUT /api/super-admin/update-plan-details → change plan limits/features

Flow 4: Tenant Monitoring
  1. GET /api/super-admin/get-tenants → dashboard with all tenants
  2. Click tenant → GET /api/super-admin/get-tenant/:tenantId → full details
  3. View invoices → GET /api/super-admin/get-tenant/:tenantId/invoices
  4. Check billing → GET /api/incubation/subscription/billing-info/:tenantId
  5. View subscription → GET /api/incubation/subscription/tenant/:tenantId

Flow 5: Invoice Management
  1. GET /api/super-admin/get-invoice → all invoices dashboard
  2. Filter by status (PAID/UNPAID/OVERDUE) or search
  3. Click tenant link → GET /api/super-admin/get-tenant/:tenantId

Flow 6: Addon Management
  1. POST /api/super-admin/create-addon → create new addon
  2. GET /api/super-admin/get-addons → list all addons
  3. GET /api/super-admin/get-addon/:addonId → addon detail
  4. PUT /api/super-admin/update-addon → update addon
  5. PATCH /api/super-admin/toggle-addon-status → enable/disable
  6. DELETE /api/super-admin/delete-addon/:addonId → delete (if no requests)

Flow 7: Addon Request Processing
  1. GET /api/super-admin/addon-requests → list all requests
  2. Filter by status (PENDING/IN_REVIEW/APPROVED/REJECTED)
  3. Click request → GET /api/super-admin/addon-requests/:requestId
  4. PATCH /api/super-admin/addon-requests/update-status → approve/reject
     (auto-sends email to requester)

Flow 8: Subscription Oversight
  1. GET /api/incubation/subscriptions → all subscriptions
  2. Filter by status (ACTIVE/CANCELED/PENDING)
  3. GET /api/incubation/subscription/tenant/:tenantId → detail view
  4. POST /api/incubation/subscription/:id/cancel → cancel if needed

Flow 9: Chat
  1. GET /api/super-admin/chats → list conversations
  2. GET /api/super-admin/chats/search-tenants → find tenant to chat with
  3. GET /api/super-admin/chats/history/:tenantId → load messages
  4. (Real-time messaging via Socket.IO — see socket-connection-guide.txt)
