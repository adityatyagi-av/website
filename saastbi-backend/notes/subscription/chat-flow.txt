1. ARCHITECTURE OVERVIEW
================================================================================

The platform has TWO independent chat systems:

SYSTEM A: SUPPORT CHAT (Tenant Admin <-> SuperAdmin)
  - Purpose: Billing inquiries, subscription help, general support
  - Protocol: REST API + Socket.IO (default namespace "/")
  - Database Models: Chat, TenantMessage
  - Access: SuperAdmin (any role) + Tenant Admin (isAdmin: true only)
  - Each tenant has exactly ONE support chat thread (Chat.tenantId is unique)

SYSTEM B: UNIVERSAL CHAT (Entity <-> Entity)
  - Purpose: Messaging between users, startups, pages, tenants
  - Protocol: REST API + Socket.IO (/chat namespace)
  - Database Models: Conversation, Message
  - Access: Ecosystem users, startup members, page members
  - Incubation portal uses CHAT SEARCH to find entities, then
    routes to universal chat for actual messaging

KEY DISTINCTION:
  - Support Chat uses Chat + TenantMessage tables
  - Universal Chat uses Conversation + Message tables
  - They are completely separate systems on separate namespaces
  - Support Chat: io("SERVER_URL")         (default namespace)
  - Universal Chat: io("SERVER_URL/chat")  (/chat namespace)

2. DATA MODELS
================================================================================

Chat (Support Chat Container):
  - id: string (cuid)
  - tenantId: string (UNIQUE — one chat per tenant)
  - lastMessage: string | null
  - lastMessageAt: DateTime | null
  - unreadCount: number (messages from tenant unread by SuperAdmin)
  - tenantUnreadCount: number (messages from SuperAdmin unread by tenant)
  - createdAt: DateTime
  - updatedAt: DateTime

TenantMessage (Individual Support Chat Message):
  - id: string (cuid)
  - chatId: string (references Chat.id)
  - senderId: string (SuperAdmin.id or IncubationUser.id)
  - senderType: "SUPER_ADMIN" | "TENANT_ADMIN"
  - content: string
  - isRead: boolean (default: false)
  - readBy: string[] (IDs of users who read this)
  - createdAt: DateTime
  - updatedAt: DateTime

Conversation (Universal Chat Container):
  - id: string (cuid)
  - participant1Id: string
  - participant1Type: "USER" | "STARTUP" | "PAGE" | "TENANT"
  - participant2Id: string
  - participant2Type: "USER" | "STARTUP" | "PAGE" | "TENANT"
  - participant1UnreadCount: number (default: 0)
  - participant2UnreadCount: number (default: 0)
  - lastMessage: string | null
  - lastMessageAt: DateTime | null
  - isBlocked: boolean (default: false)
  - createdAt: DateTime
  - updatedAt: DateTime

Message (Universal Chat Message):
  - id: string (cuid)
  - conversationId: string (references Conversation.id)
  - senderId: string
  - senderType: "USER" | "STARTUP" | "PAGE" | "TENANT"
  - content: string | null
  - messageType: "TEXT" | "IMAGE" | "FILE" (default: "TEXT")
  - attachments: JSON | null
  - replyToId: string | null (references Message.id)
  - isRead: boolean (default: false)
  - isEdited: boolean (default: false)
  - isDeleted: boolean (default: false)
  - createdAt: DateTime
  - updatedAt: DateTime

6. SOCKET.IO REAL-TIME CHAT
================================================================================

Server URL: Same as HTTP server (e.g., http://localhost:7000)
Transport: WebSocket with polling fallback
CORS: Allowed from process.env.CLIENT_URL (or "*" if not set)

--------------------------------------------------------------------------------
6a. SUPPORT CHAT SOCKET (Default Namespace "/")
--------------------------------------------------------------------------------

CONNECTING AS SUPER ADMIN:

  const socket = io("SERVER_URL", {
    auth: { token: "<superadmin_jwt_token>" }
  });

  Token Requirements:
    - SuperAdmin JWT with payload: { id, role }
    - role must be one of: "ADMIN", "MARKETING", "MANAGER", "RELATIONSHIP_MANAGER"

  On Connect:
    - socket.userType = "SUPER_ADMIN"
    - Auto-joins room "super-admins"
    - Receives messages from ALL tenants

CONNECTING AS TENANT ADMIN:

  const socket = io("SERVER_URL", {
    auth: { token: "<incubation_user_jwt_token>" }
  });

  Token Requirements:
    - IncubationUser JWT with payload: { id }
    - Token must NOT have a role field (that identifies SuperAdmin)
    - User must have isAdmin: true
    - Non-admin users are rejected

  On Connect:
    - socket.userType = "TENANT_ADMIN"
    - Auto-joins room "tenant-{tenantId}"
    - Receives messages only for YOUR tenant

EMIT EVENTS (Frontend -> Server):

  1. send-message
     {
       "content": "Message text",
       "tenantId": "cuid_tenant_123"   // REQUIRED for SuperAdmin
                                        // Auto-resolved for TenantAdmin
     }
     - Chat record auto-creates on first message
     - Response: "message-sent" event to sender

  2. mark-as-read
     {
       "chatId": "cuid_chat_1"
     }
     - Only SuperAdmin can mark tenant messages as read
     - Adds SuperAdmin ID to readBy array
     - Resets Chat.unreadCount to 0
     - Response: "messages-read" event to sender

  3. typing
     {
       "tenantId": "cuid_tenant_123",   // REQUIRED for SuperAdmin
       "isTyping": true                  // Auto-resolved for TenantAdmin
     }
     - SuperAdmin typing -> broadcasts to "tenant-{tenantId}" room
     - TenantAdmin typing -> broadcasts to "super-admins" room

LISTEN EVENTS (Server -> Frontend):

  1. message-sent
     Full message object with sender and chat details.
     Sent to SENDER as confirmation.

  2. new-message
     Same payload as message-sent.
     Sent to RECIPIENTS (not sender):
       - TenantAdmin sends -> "super-admins" room
       - SuperAdmin sends -> "tenant-{tenantId}" room

  3. messages-read
     { "chatId": "cuid_chat_1" }
     Confirmation after mark-as-read.

  4. user-typing
     {
       "tenantId": "cuid_tenant_123",
       "userName": "Admin User",
       "isTyping": true
     }

  5. error
     { "message": "Error description" }

MESSAGE PAYLOAD STRUCTURE (for message-sent and new-message):
{
  "id": "cuid_msg_new",
  "chatId": "cuid_chat_1",
  "senderId": "cuid_user_id",
  "senderType": "SUPER_ADMIN" | "TENANT_ADMIN",
  "content": "Message text",
  "isRead": false,
  "readBy": [],
  "createdAt": "2026-03-29T10:00:00Z",
  "updatedAt": "2026-03-29T10:00:00Z",
  "sender": {
    "id": "cuid_user_id",
    "name": "User Name",
    "type": "SUPER_ADMIN" | "TENANT_ADMIN"
  },
  "chat": {
    "id": "cuid_chat_1",
    "tenantId": "cuid_tenant_123",
    "tenant": {
      "id": "cuid_tenant_123",
      "organizationName": "Innovation Hub",
      "tenantKey": "innovation-hub"
    }
  }
}

--------------------------------------------------------------------------------
6b. UNIVERSAL CHAT SOCKET (/chat Namespace)
--------------------------------------------------------------------------------

This is used when the incubation portal user selects an entity from
chat search (section 5) and starts a conversation.

CONNECTING:

  const socket = io("SERVER_URL/chat", {
    auth: { token: "<ecosystem_user_jwt_token>" }
  });

  Token Requirements:
    - User JWT (ecosystem user from User table)
    - Token payload: { id }
    - decoded.id looked up in User table

  On Connect:
    - Auto-joins room "user:{userId}"
    - Receives "conversation-updated" events even without joining conversations

EMIT EVENTS:

  1. join-conversation (conversationId as string)
     - Must be participant1 or participant2
     - Joins room "conversation:{conversationId}"
     - Response: "joined-conversation" { conversationId }

  2. leave-conversation (conversationId as string)
     - Leaves room "conversation:{conversationId}"

  3. send-message
     {
       "conversationId": "cuid_conv_1",
       "content": "Hello!",
       "messageType": "TEXT",        // Optional, default: "TEXT"
       "replyToId": null,            // Optional
       "attachments": null,          // Optional
       "startupId": "cuid_s_1",     // Optional: send as STARTUP entity
       "pageId": "cuid_p_1"         // Optional: send as PAGE entity
     }

  4. typing
     { "conversationId": "cuid_conv_1", "isTyping": true }

  5. mark-read
     {
       "conversationId": "cuid_conv_1",
       "startupId": "cuid_s_1"      // Optional: mark for STARTUP entity
     }

  6. delete-message
     {
       "messageId": "cuid_msg_1",
       "conversationId": "cuid_conv_1",
       "startupId": "cuid_s_1"      // Optional: if sent as STARTUP
     }

  7. edit-message
     {
       "messageId": "cuid_msg_1",
       "conversationId": "cuid_conv_1",
       "content": "Updated text",
       "startupId": "cuid_s_1"      // Optional: if sent as STARTUP
     }
     - Only within 5 minutes of sending

LISTEN EVENTS:

  1. joined-conversation — { conversationId }
  2. message-sent — Full message (to sender only)
  3. new-message — Full message (to recipients in conversation room)
  4. conversation-updated — { conversationId, lastMessage } (to user rooms)
  5. user-typing — { conversationId, userId, userName, isTyping }
  6. messages-read — { conversationId } (confirmation to sender)
  7. participant-read — { conversationId, userId, readAt } (read receipts)
  8. message-deleted — { messageId, conversationId }
  9. message-edited — { messageId, conversationId, content, editedAt }
  10. error — { message }

7. FRONTEND FLOWS — SUPER ADMIN
================================================================================

Flow 1: View All Support Chats
  1. GET /api/super-admin/chats?page=1&limit=20
  2. Render chat list sorted by lastMessageAt descending
  3. Show unreadCount badge on each chat item
  4. Click a chat → Flow 2

Flow 2: Open a Support Chat with Tenant
  1. GET /api/super-admin/chats/history/:tenantId?page=1&limit=50
  2. Render messages:
     - Right-align SUPER_ADMIN messages (yours)
     - Left-align TENANT_ADMIN messages (tenant's)
  3. Connect Socket.IO to default namespace if not connected
  4. Emit "mark-as-read" { chatId } to clear unread count
  5. Listen "new-message" for this chat's tenantId
  6. Listen "user-typing" for typing indicator

Flow 3: Send a Support Message
  1. User types message
  2. Emit socket "send-message" { content, tenantId }
  3. Listen "message-sent" → append to chat UI
  4. Other SuperAdmins also receive "new-message"

Flow 4: Search & Start New Chat with Tenant
  1. GET /api/super-admin/chats/search-tenants?search=query
  2. Show results with hasChat indicator
  3. hasChat: true → Navigate to existing chat (Flow 2)
  4. hasChat: false → Open empty chat window, send first message via socket
     (Chat record auto-creates on first message)

Flow 5: View Tenant Billing Context in Chat
  When a tenant messages about billing, the SuperAdmin can reference:
  - Tenant's subscription status from the tenant detail page
  - No specific billing endpoint is needed IN the chat — use tenant
    management APIs to look up billing data alongside the chat

     - Right-align TENANT_ADMIN messages (your messages)
     - Left-align SUPER_ADMIN messages (support messages)
  3. POST /api/incubation-portal/support-chat/mark-read (mark as read)
  4. Connect Socket.IO to default namespace
  5. Listen "new-message" → append incoming SuperAdmin messages
  6. Listen "user-typing" → show "Support is typing..."

Flow 3: Send Support Message
  1. User types message
  2. Emit socket "send-message" { content }
     (tenantId auto-resolved from token — no need to send it)
  3. Listen "message-sent" → append to chat UI
  4. SuperAdmin receives "new-message" on their end

Flow 4: Ask About Billing via Support Chat
  1. Tenant admin opens support chat (Flow 2)
  2. Sends billing inquiry via socket "send-message"
  3. SuperAdmin sees the message in their chat list
  4. SuperAdmin can look up billing/subscription info and respond
  5. Meanwhile, tenant admin can check their own billing via:
     - GET /api/incubation-portal/billing (portal billing dashboard)
     - GET /api/incubation/subscription/billing-info/:tenantId

Flow 5: Search for Entities to Chat With
  1. Navigate to "Chat" section in incubation portal
  2. GET /api/incubation-portal/chat/search?q=&filter=
  3. Display results with filter tabs: All | Tenants | Pages | Startups | Mentors
  4. Associated mentors and enrolled startups appear first (higher proximityScore)
  5. User selects an entity → create/get conversation via universal chat
  6. Connect to Socket.IO /chat namespace for real-time messaging

Flow 6: Universal Chat from Incubation Portal
  1. After selecting entity from search (Flow 5):
  2. Create or get existing conversation via REST API
  3. Connect to io("SERVER_URL/chat") with ecosystem user token
  4. Emit "join-conversation" with conversationId
  5. Emit "send-message" with conversationId and content
  6. Listen "new-message" for incoming messages
  7. Listen "conversation-updated" for sidebar updates

Flow 7: Real-time Support Chat Updates
  1. Socket connected to default namespace
  2. Auto-joins "tenant-{tenantId}" room
  3. Listen "new-message" → notification + update chat
  4. Listen "user-typing" → show "Support is typing..."
  5. Emit "typing" when typing

9. CHAT + BILLING INTEGRATION FLOW
================================================================================

This section describes how chat and billing features work together in the
incubation portal UI.

SCENARIO: Tenant Admin Checks Billing & Contacts Support

Step 1 — View Billing Dashboard:
  GET /api/incubation-portal/billing
  Response includes: plan details, next billing date, days until renewal,
  recent payment history.

Step 2 — Make a Payment (if needed):
  POST /api/incubation-portal/billing/create-order
  Returns Razorpay order ID → open Razorpay checkout UI

Step 3 — Verify Payment:
  POST /api/incubation-portal/billing/verify-payment
  Body: { razorpayOrderId, razorpayPaymentId, razorpaySignature }
  Returns: invoice + billing record + updated subscription dates

Step 4 — Contact Support About Billing Issue:
  GET /api/incubation-portal/support-chat (check if chat exists)
  Connect to Socket.IO default namespace
  Emit "send-message" { content: "I have a billing question..." }
  SuperAdmin receives the message and can look up the tenant's billing info

SCENARIO: SuperAdmin Reviews Billing & Responds in Chat

Step 1 — View Tenant Chats:
  GET /api/super-admin/chats?page=1&limit=20

Step 2 — Open Specific Tenant Chat:
  GET /api/super-admin/chats/history/:tenantId

Step 3 — Look Up Tenant Billing (in separate view):
  SuperAdmin uses tenant management to view subscription/billing details

Step 4 — Respond to Tenant:
  Emit socket "send-message" { content: "Your billing is...", tenantId }

SCENARIO: Renewal Reminder Triggers Support Chat

Step 1 — Cron job sends renewal reminder email (2 days before monthly,
  20 days before yearly, 2 days before weekly)

Step 2 — Tenant admin receives email with billing info

Step 3 — If tenant has questions, opens support chat:
  GET /api/incubation-portal/support-chat
  Connect Socket.IO → send billing inquiry

10. ERROR REFERENCE
================================================================================

Support Chat Errors:
  - 403 "Only super admins can access all chats"
    Cause: Non-SuperAdmin token used on SuperAdmin endpoint
  - 403 "This endpoint is for tenant admins only"
    Cause: Non-admin IncubationUser accessing portal chat endpoints
  - 400 "Search query is required"
    Cause: Missing ?search= parameter on search-tenants endpoint

Socket.IO Auth Errors (connect_error event):
  - "Authentication error: No token provided"
    Cause: No token in socket.handshake.auth or headers
  - "Authentication error: User not found"
    Cause: Token ID doesn't match any database record
  - "Authentication error: Only tenant admins can chat"
    Cause: IncubationUser with isAdmin: false
  - "Authentication error: jwt expired"
    Cause: Expired token — refresh and reconnect
  - "Authentication error: invalid signature"
    Cause: Token signed with wrong secret

Chat Search Errors:
  - 400 "tenantKey header is required"
    Cause: Missing tenantKey header on incubation portal chat search
  - 404 "Tenant not found"
    Cause: Invalid tenantKey
  - 403 "Only tenant admins can access chat search"
    Cause: User is not an active admin for this tenant
  - 403 "Incubation user not found"
    Cause: Authenticated user has no IncubationUser record

Universal Chat Socket Errors:
  - "You are not a participant in this conversation"
    Cause: Trying to join a conversation you're not part of
  - "You are not a member of this startup"
    Cause: Invalid startupId or inactive membership
  - "You are not a member of this page"
    Cause: Invalid pageId or no membership
  - "Message not found or you cannot delete this message"
    Cause: Wrong senderId for edit/delete (forgot startupId/pageId)
  - "Cannot edit message after 5 minutes"
    Cause: Edit window expired
