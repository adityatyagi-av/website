================================================================================
FUNDING - COMPLETE END-TO-END FLOW (Incubation Portal + Startup Portal)
================================================================================

This document explains the entire funding lifecycle of the SaaSTBI platform:
how money enters the system, how it is attached to programs and batches, how
startups receive it (admin-driven) or request it (startup-driven), how the
state machine evolves, and how every model in the Prisma schema participates.

Read this first before reading the incubation-perspective.txt or
startup-perspective.txt files.

--------------------------------------------------------------------------------
1. BIG-PICTURE MENTAL MODEL
--------------------------------------------------------------------------------

Think of funding as four layers, top to bottom:

    Layer 1  TenantFundingSource          (money a tenant has in the bank)
                  |
                  |  split into
                  v
    Layer 2  ProgramFundingAllocation     (earmarked for a program / batch)
                  |
                  |  paid out as
                  v
    Layer 3  FundingDisbursement          (actual payment to a startup)
                  |
                  |  accumulates into
                  v
    Layer 4  StartupProgramAssociation    (per-startup running totals)
             (.approvedFundingAmount, .totalDisbursedAmount)

In parallel, a startup may create a "pull request" style record called
StartupFundingRequest that, once approved by an incubation admin, auto-creates
a Layer 3 FundingDisbursement and is linked 1-to-1 to it via `disbursementId`.

Every write action is recorded in FundingHistory (immutable audit log).

--------------------------------------------------------------------------------
2. SCHEMA (prisma/schema.prisma) - HOW THE MODELS CONNECT
--------------------------------------------------------------------------------

TenantFundingSource
  - tenantId, sourceName, sourceType (GOVT_GRANT | CSR | INTERNAL | INVESTOR
    | DONATION | SPONSORED | OTHER)
  - totalAmount, allocatedAmount, currency
  - isActive (soft-delete flag), receivedDate, expiryDate, reference, notes
  - allocations: ProgramFundingAllocation[]
  - documents:   FundingDocument[] (ownerType SOURCE)

ProgramFundingAllocation
  - programId, fundingSourceId, batchId?, allocatedAmount, currency
  - UNIQUE (programId, fundingSourceId, batchId)
  - batchId is OPTIONAL: an allocation can live at program-level OR batch-level
  - program: Program, batch: ProgramBatch?, fundingSource: TenantFundingSource

Program (funding-specific fields)
  - totalFundingAmount, fundingType (EQUITY | GRANT | DEBT | CONVERTIBLE_NOTE
    | MIXED), fundingCurrency, isFundingAvailable
  - fundingAllocations[], fundingDisbursements[], fundingRequests[]

ProgramBatch (mirrors program fields at batch level)
  - totalFundingAmount, fundingType, fundingCurrency, isFundingAvailable
  - fundingAllocations[], fundingDisbursements[], fundingRequests[]

StartupApplication
  - requestedFundingAmount, fundingPurpose (what the startup said in the form)
  - disbursements[], fundingRequests[]

StartupProgramAssociation
  - startupId, programId, batchId?, status (ACTIVE | COMPLETED | OFFBOARDED
    | REMOVED), onboardedAt, offboardedAt, isActive
  - approvedFundingAmount  -> the CAP of money this startup can receive in
                              this program/batch
  - totalDisbursedAmount   -> running sum of COMPLETED disbursements
  - UNIQUE (startupId, programId, batchId)

FundingDisbursement
  - programId, applicationId, startupId, tenantId, batchId?
  - amount, currency
  - disbursementType (LUMP_SUM | TRANCHE | MILESTONE_BASED)
  - status (PENDING | APPROVED | PROCESSING | COMPLETED | FAILED | CANCELLED)
  - trancheNumber? (auto-increment per application when type=TRANCHE)
  - milestoneName?, reference? (UTR), disbursedById?, disbursedAt?, notes
  - documents: FundingDocument[] (ownerType DISBURSEMENT)
  - fundingRequest?: StartupFundingRequest (back-relation)

StartupFundingRequest
  - applicationId, startupId, programId, tenantId, batchId?
  - requestedAmount, currency, note, docUrl, docName
  - status (PENDING | APPROVED | REJECTED | CANCELLED)
  - reviewedById, reviewNote, approvedAmount
  - disbursementId UNIQUE -> one-to-one link with FundingDisbursement
    (set when the request is approved)

FundingDocument
  - ownerType (SOURCE | DISBURSEMENT), fundingSourceId?, disbursementId?
  - fileName, fileUrl, fileType, description, uploadedById

FundingHistory (audit log, append only)
  - tenantId, action (see enum), entityType, entityId, performedById
  - oldValue: Json, newValue: Json, notes, createdAt

Enum FundingActionType:
  SOURCE_CREATED, SOURCE_UPDATED, SOURCE_DEACTIVATED,
  FUNDING_ALLOCATED, ALLOCATION_UPDATED, ALLOCATION_REMOVED,
  DISBURSEMENT_CREATED, DISBURSEMENT_APPROVED, DISBURSEMENT_COMPLETED,
  DISBURSEMENT_FAILED, DISBURSEMENT_CANCELLED,
  REQUEST_CREATED, REQUEST_APPROVED, REQUEST_REJECTED, REQUEST_CANCELLED

--------------------------------------------------------------------------------
3. HOW FUNDING IS ATTACHED TO PROGRAMS AND BATCHES
--------------------------------------------------------------------------------

Funding is attached in TWO places, and both are honored:

  (a) Program-level earmark (Program.totalFundingAmount, fundingType, ...)
      This is a "declared capacity" set at program creation time. It tells the
      platform how much the program INTENDS to distribute.

  (b) Batch-level earmark (ProgramBatch.totalFundingAmount, ...)
      Same fields at batch granularity. When a program runs multiple batches
      (e.g. Cohort-1, Cohort-2), each batch can have its own pot.

But the ACTUAL money that flows comes from TenantFundingSource via
ProgramFundingAllocation. The allocation binds a source to:

  - a program (programId, required), AND
  - optionally a batch (batchId, nullable).

Because the unique key is (programId, fundingSourceId, batchId):

  - You can allocate the same source to the same program MULTIPLE times IF
    the batchId differs.
  - If you call "allocate" on an existing (program, source, batch) triple,
    the service INCREMENTS the existing row (does not create a second row
    and does not overwrite). This is intentional top-up semantics.

Because batchId is nullable, and Prisma treats NULL as distinct in unique
constraints, the service uses findFirst (not findUnique) whenever it looks
up allocations/associations by (program, batch[, source|startup]).

--------------------------------------------------------------------------------
4. TWO FLOWS THAT END IN THE SAME PLACE
--------------------------------------------------------------------------------

There are two entry points to create a FundingDisbursement:

  FLOW A - Admin-initiated (push)
    Incubation admin decides to disburse -> creates FundingDisbursement
    directly (status=PENDING) -> advances through state machine.

  FLOW B - Startup-initiated (pull)
    Startup creates a StartupFundingRequest (status=PENDING) -> admin reviews
    -> on approve, service auto-creates FundingDisbursement (status=PENDING)
    and links it back via StartupFundingRequest.disbursementId. From here,
    flow is identical to FLOW A.

Both flows converge on the same FundingDisbursement state machine, the same
accounting in StartupProgramAssociation, and the same FundingHistory audit
entries.

--------------------------------------------------------------------------------
5. THE DISBURSEMENT STATE MACHINE
--------------------------------------------------------------------------------

Valid transitions (enforced in fundingService.updateDisbursementStatus):

    PENDING    -> APPROVED | CANCELLED
    APPROVED   -> PROCESSING | CANCELLED
    PROCESSING -> COMPLETED | FAILED
    FAILED     -> PROCESSING | CANCELLED
    COMPLETED  -> CANCELLED                (reversal / claw-back)
    CANCELLED  -> (terminal)

Side effects:

  - Entering COMPLETED:
      association.totalDisbursedAmount += disbursement.amount
      disbursement.disbursedAt = now()
      FundingHistory action: DISBURSEMENT_COMPLETED
  - Leaving COMPLETED (only COMPLETED -> CANCELLED is allowed):
      association.totalDisbursedAmount -= disbursement.amount   (reversal)
      FundingHistory action: DISBURSEMENT_CANCELLED

--------------------------------------------------------------------------------
6. END-TO-END WALK-THROUGH (HAPPY PATH, ADMIN-INITIATED)
--------------------------------------------------------------------------------

Pre-conditions:
  - Tenant exists.
  - Incubation admin user has MODULE_KEYS.FUNDING access.
  - A program (and optionally a batch) exists.
  - The startup has applied via StartupApplication and has been ONBOARDED
    (StartupApplication.status = ONBOARDED) - this creates a
    StartupProgramAssociation with approvedFundingAmount set on onboarding.

Step 1. Create a funding source
  POST /api/incubation-portal/funding/sources
  -> TenantFundingSource row created (isActive=true, allocatedAmount=0)
  -> FundingHistory: SOURCE_CREATED

Step 2. Allocate money from the source to the program (optionally to a batch)
  POST /api/incubation-portal/funding/program/:programId/allocations
    body: { fundingSourceId, batchId?, allocatedAmount, currency, notes }
  -> Validates: allocatedAmount <= source.totalAmount - source.allocatedAmount
  -> If (programId, fundingSourceId, batchId) exists: INCREMENT
     Else: create new ProgramFundingAllocation
  -> source.allocatedAmount += allocatedAmount
  -> FundingHistory: FUNDING_ALLOCATED

Step 3. Create a disbursement for a specific onboarded startup
  POST /api/incubation-portal/funding/program/:programId/disbursements
    body: { applicationId, startupId, amount, currency, disbursementType,
            milestoneName?, reference?, notes?, documents? }
  -> Validates: application.status == ONBOARDED and belongs to program/tenant.
  -> Validates: association exists (findFirst with explicit batchId).
  -> Validates: if association.approvedFundingAmount is set,
                amount <= approvedFundingAmount - totalDisbursedAmount.
  -> For TRANCHE: trancheNumber = maxExistingTranche(application) + 1.
  -> Disbursement row created with status=PENDING.
  -> FundingDocument rows (ownerType=DISBURSEMENT) created for attachments.
  -> FundingHistory: DISBURSEMENT_CREATED

Step 4. Walk status: PENDING -> APPROVED -> PROCESSING -> COMPLETED
  PATCH /api/incubation-portal/funding/disbursements/:disbursementId/status
    body: { newStatus, reference?, notes? }
  Each transition validated against the state machine.
  On COMPLETED:
    - association.totalDisbursedAmount += amount
    - disbursedAt = now()
  FundingHistory entries: DISBURSEMENT_APPROVED, then DISBURSEMENT_COMPLETED.

Step 5. (Optional) Reverse a mistaken completion
  PATCH .../status  with newStatus=CANCELLED on a COMPLETED row
  -> association.totalDisbursedAmount -= amount
  -> FundingHistory: DISBURSEMENT_CANCELLED

Step 6. Report
  GET /api/incubation-portal/funding/program/:programId/portfolio
    per-program totals + per-startup breakdown
  GET /api/incubation-portal/funding/overview
    tenant-wide KPIs (sources, allocated, disbursed, pending, by source type)

--------------------------------------------------------------------------------
7. END-TO-END WALK-THROUGH (STARTUP-INITIATED)
--------------------------------------------------------------------------------

Pre-conditions: Same as above (application must be ONBOARDED, the startup
user must have access to the startup via verifyUserStartupAccess).

Step 1. Startup creates the request
  POST /api/startup/application/:applicationId/funding-request
    body: { startupId, requestedAmount, currency, note?, docUrl?, docName? }
  -> Validates ONBOARDED, association exists, and
     requestedAmount <= approvedFundingAmount - totalDisbursedAmount.
  -> Row created with status=PENDING. tenantId and batchId copied from
     the application.
  -> FundingHistory: REQUEST_CREATED

Step 2. Incubation admin reviews
  GET /api/incubation-portal/funding/requests?status=PENDING&programId=...

Step 3a. Admin approves
  PATCH /api/incubation-portal/funding/requests/:requestId/approve
    body: { approvedAmount?, disbursementType?, currency?, reference?, notes? }
  -> Validates: request.status == PENDING; approvedAmount (defaults to
     requestedAmount) <= remaining balance.
  -> Service AUTO-CREATES a FundingDisbursement (status=PENDING).
     For TRANCHE type, trancheNumber auto-increments.
  -> request.status = APPROVED, reviewedById, approvedAmount, disbursementId.
  -> FundingHistory: REQUEST_APPROVED (and implicitly DISBURSEMENT_CREATED
     for the new disbursement).
  From here the disbursement follows the exact same state machine as
  FLOW A (APPROVED -> PROCESSING -> COMPLETED).

Step 3b. Admin rejects
  PATCH /api/incubation-portal/funding/requests/:requestId/reject
    body: { reviewNote }
  -> request.status = REJECTED; FundingHistory: REQUEST_REJECTED.

Step 3c. Startup cancels while still PENDING
  PATCH /api/startup/funding-request/:requestId/cancel
    body: { startupId }
  -> Only allowed when request.status == PENDING.
  -> request.status = CANCELLED; no disbursement exists to touch.

--------------------------------------------------------------------------------
8. ACCOUNTING INVARIANTS
--------------------------------------------------------------------------------

The service layer maintains these invariants:

  (I1) source.allocatedAmount == sum(allocations for that source)
  (I2) source.allocatedAmount <= source.totalAmount
  (I3) source.isActive == false only if source.allocatedAmount == 0
  (I4) association.totalDisbursedAmount == sum(disbursements with
        status=COMPLETED for that (startup, program, batch))
  (I5) if association.approvedFundingAmount is set,
        association.totalDisbursedAmount + sum(PENDING/APPROVED/PROCESSING
        disbursement amounts) + sum(PENDING request amounts)
          <= association.approvedFundingAmount
       (the service checks this at every write point)
  (I6) StartupFundingRequest.disbursementId is set iff status == APPROVED

--------------------------------------------------------------------------------
9. AUTHN / AUTHZ SUMMARY
--------------------------------------------------------------------------------

Incubation portal (admin side):
  - Auth: authenticatePortal (portal bearer token) + tenantkey header.
  - Authz: requireAccessByMethod checking MODULE_KEYS.FUNDING.
  - Tenant scoping: resolveTenantId(req) gates every query.

Startup portal (startup side):
  - Auth: authenticate (user bearer token).
  - Authz: verifyUserStartupAccess(userId, startupId) checked inside service
    functions; the user must belong to the startup.
  - Only ONBOARDED applications may create requests.
