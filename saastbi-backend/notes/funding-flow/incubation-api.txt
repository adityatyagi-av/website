================================================================================
FUNDING API - INCUBATION PORTAL (DETAILED FRONTEND REFERENCE)
================================================================================

Audience: frontend engineers (or an LLM generating UI code) building the
incubation-portal funding screens. Every endpoint in this document is
derived from:

  src/routes/incubation/portal/funding.route.js
  src/controllers/incubation/portal/funding.controller.js
  src/services/incubation/portal/funding.service.js
  prisma/schema.prisma (models TenantFundingSource, ProgramFundingAllocation,
                       FundingDisbursement, FundingDocument, FundingHistory,
                       StartupFundingRequest, StartupProgramAssociation,
                       Program, ProgramBatch, StartupApplication)

Global conventions
------------------
- Base URL prefix: /api/incubation-portal
- Auth header:     Authorization: Bearer <portal token>     (authenticatePortal)
- Tenant header:   tenantkey: <tenant key>                  (resolveTenantId)
- Access gate:     requireAccessByMethod(MODULE_KEYS.FUNDING)
                   All routes below are gated on the FUNDING module.
- Response envelope (apiResponse.sendSuccess):
    {
      "success": true,
      "message": "<human message>",
      "data":    <payload described per endpoint>
    }
- Errors come through errorHandler as:
    { "success": false, "message": "<reason>", "statusCode": 4xx }
- Amount fields are Float (INR by default, pass a different `currency` for
  other currencies). The server does NOT convert currencies; choose per
  funding source and keep the chain consistent.
- All CUIDs. All timestamps are ISO-8601 strings.

Enums (names exactly as stored)
-------------------------------
FundingSourceType:     GOVT_GRANT | CSR | INTERNAL | INVESTOR | DONATION
                       | SPONSORED | OTHER
ProgramFundingType:    EQUITY | GRANT | DEBT | CONVERTIBLE_NOTE | MIXED
DisbursementType:      LUMP_SUM | TRANCHE | MILESTONE_BASED
DisbursementStatus:    PENDING | APPROVED | PROCESSING | COMPLETED | FAILED
                       | CANCELLED
FundingRequestStatus:  PENDING | APPROVED | REJECTED | CANCELLED
FundingDocOwnerType:   SOURCE | DISBURSEMENT
FundingActionType:     SOURCE_CREATED | SOURCE_UPDATED | SOURCE_DEACTIVATED
                       | FUNDING_ALLOCATED | ALLOCATION_UPDATED | ALLOCATION_REMOVED
                       | DISBURSEMENT_CREATED | DISBURSEMENT_APPROVED
                       | DISBURSEMENT_PROCESSING | DISBURSEMENT_COMPLETED
                       | DISBURSEMENT_FAILED | DISBURSEMENT_CANCELLED
                       | REQUEST_CREATED | REQUEST_APPROVED | REQUEST_REJECTED
                       | REQUEST_CANCELLED

Valid disbursement status transitions
-------------------------------------
  PENDING    -> APPROVED | CANCELLED
  APPROVED   -> PROCESSING | CANCELLED
  PROCESSING -> COMPLETED | FAILED
  FAILED     -> PROCESSING | CANCELLED
  COMPLETED  -> CANCELLED    (reversal; decrements totalDisbursedAmount)
  CANCELLED  -> (terminal)


================================================================================
SECTION 1 - FUNDING SOURCES
================================================================================
A TenantFundingSource is a pool of money the tenant has received
(grant/CSR/investor/etc.). Sources are allocated to programs/batches, then
disbursed to onboarded startups.

--------------------------------------------------------------------------------
1.1  POST /funding/sources - Create source
--------------------------------------------------------------------------------
Body (application/json):
  {
    "sourceName":   "DST NIDHI Grant",          // string, REQUIRED
    "sourceType":   "GOVT_GRANT",               // FundingSourceType, REQUIRED
    "totalAmount":  50000000,                   // float > 0, REQUIRED
    "currency":     "INR",                      // optional, default "INR"
    "receivedDate": "2026-01-01",               // optional ISO date
    "expiryDate":   "2027-12-31",               // optional ISO date
    "reference":    "DST/NIDHI/2026-001",       // optional string
    "notes":        "3-year grant",             // optional string
    "documents": [                              // optional array, attached as SOURCE
      {
        "fileName":    "sanction.pdf",
        "fileUrl":     "https://s3...",          // pre-uploaded S3 URL
        "fileType":    "application/pdf",        // optional
        "description": "Sanction letter"         // optional
      }
    ]
  }

Validation errors (400):
  - "sourceName is required"
  - "sourceType is required"
  - "totalAmount must be a positive number"

Response data:
  TenantFundingSource with `documents: FundingDocument[]` included.
  Important fields:
    id, tenantId, sourceName, sourceType, totalAmount, allocatedAmount (=0
    at creation), currency, receivedDate, expiryDate, reference, notes,
    isActive (=true), createdAt, updatedAt.

Audit:
  FundingHistory row with action=SOURCE_CREATED.

--------------------------------------------------------------------------------
1.2  GET /funding/sources - List sources for current tenant
--------------------------------------------------------------------------------
Query params: none.

Response data (array):
  Only sources with isActive=true, newest first.
  Each item includes:
    - documents: FundingDocument[] (ownerType=SOURCE)
    - allocations: ProgramFundingAllocation[] with
        program: { id, title }
  Use `source.allocatedAmount` vs `source.totalAmount` to show remaining pool.

--------------------------------------------------------------------------------
1.3  GET /funding/sources/:sourceId - Single source
--------------------------------------------------------------------------------
Path params:
  sourceId (CUID)

Notes:
  Returns even if isActive=false. Still enforces tenant ownership.

Response data:
  TenantFundingSource with `documents` and `allocations` (with program).

Errors:
  404 "Funding source not found" (missing or cross-tenant).

--------------------------------------------------------------------------------
1.4  PATCH /funding/sources/:sourceId - Update source
--------------------------------------------------------------------------------
Path params: sourceId (CUID).

Body: partial; any subset of
  {
    "sourceName":   "Updated name",
    "sourceType":   "CSR",
    "totalAmount":  60000000,
    "currency":     "INR",
    "receivedDate": "2026-02-01",
    "expiryDate":   "2028-12-31",
    "reference":    "NEW-REF",
    "notes":        "Updated notes",
    "documents":    [ ...FundingDocument-like objects to APPEND... ]
  }

Rules:
  - `totalAmount` cannot be reduced below current `allocatedAmount`
    -> 400 "Cannot reduce total amount below already allocated amount (<n>)"
  - `documents` in the body are APPENDED; nothing is deleted from existing
    documents. To delete a document, use a future delete endpoint (not yet
    exposed - TODO).

Response data:
  Updated TenantFundingSource with `documents` included.

Audit:
  FundingHistory action=SOURCE_UPDATED.

--------------------------------------------------------------------------------
1.5  DELETE /funding/sources/:sourceId - Soft-delete (deactivate)
--------------------------------------------------------------------------------
Rules:
  - Fails if `allocatedAmount > 0`:
    -> 400 "Cannot delete a funding source that has active allocations.
       Remove allocations first."
  - Sets isActive=false; row is kept for audit.

Response data:
  { "message": "Funding source deactivated" }

Audit:
  FundingHistory action=SOURCE_DEACTIVATED.

SECTION 2 - PROGRAM FUNDING ALLOCATIONS
================================================================================
A ProgramFundingAllocation pins an amount FROM a source TO a program,
optionally scoped to a batch. Unique key:
  (programId, fundingSourceId, batchId)
batchId is nullable (Prisma treats NULL as distinct). A program-level
allocation (batchId=null) acts as a fallback pool for any batch under
the program when no batch-level allocation exists for that batch.

--------------------------------------------------------------------------------
2.1  POST /funding/program/:programId/allocations - Allocate (with top-up)
--------------------------------------------------------------------------------
Path params: programId (CUID).
Body:
  {
    "fundingSourceId": "cuid",       // REQUIRED, must be isActive
    "batchId":         "cuid" | null, // optional
    "allocatedAmount": 5000000,      // float > 0, REQUIRED
    "currency":        "INR",        // optional, default "INR"
    "notes":           "Q2 cohort"   // optional
  }

Semantics:
  - If a row with (programId, fundingSourceId, batchId) already exists,
    the amount is INCREMENTED on the same row (top-up). The `notes` field
    in the body overwrites the notes on the existing row when provided.
  - Else, a new ProgramFundingAllocation is created.
  - `source.allocatedAmount += allocatedAmount` on the funding source.

Errors:
  - 404 "Funding source not found or inactive"
  - 404 "Program not found"
  - 400 "Allocation exceeds available funds. Available: <x>, Requested: <y>"
    where available = source.totalAmount - source.allocatedAmount.

Response data:
  The newly-created or incremented ProgramFundingAllocation row (not
  wrapped). Frontend usually re-fetches 2.2 right after this.

Audit:
  FundingHistory action=FUNDING_ALLOCATED.

--------------------------------------------------------------------------------
2.2  GET /funding/program/:programId/allocations - List
--------------------------------------------------------------------------------
Returns all allocations on the program (across all batches including null).
Each item embeds:
  fundingSource: { id, sourceName, sourceType, totalAmount,
                   allocatedAmount, currency }

Frontend tip: group by batchId (null = program-level) to render two
sections.

--------------------------------------------------------------------------------
2.3  PATCH /funding/program/:programId/allocations/:allocationId
--------------------------------------------------------------------------------
Body:
  {
    "allocatedAmount": 6000000,   // > 0, REQUIRED
    "notes":           "revised"  // optional
  }

Semantics:
  diff = newAmount - existingAmount.
  If diff > 0: `diff <= (source.totalAmount - source.allocatedAmount)`
              else -> 400 "Increase exceeds available funds. Available
              from source: <x>".
  Always: `source.allocatedAmount += diff` (supports negative).

Response data:
  Updated ProgramFundingAllocation.

Audit:
  FundingHistory action=ALLOCATION_UPDATED.

--------------------------------------------------------------------------------
2.4  DELETE /funding/program/:programId/allocations/:allocationId
--------------------------------------------------------------------------------
Removes an allocation row and refunds the source.
Response data: { "message": "Allocation removed" }
Audit: FundingHistory action=ALLOCATION_REMOVED.

SECTION 3 - DISBURSEMENTS (ADMIN-INITIATED)
================================================================================
A FundingDisbursement is the actual money movement to a single startup's
application. It follows a strict state machine (see top of document).

Accounting enforcement at creation time:
  (a) Application must be ONBOARDED and belong to this program/tenant.
  (b) StartupProgramAssociation must exist for
      (startupId, programId, batchId=application.batchId || null).
  (c) If association.approvedFundingAmount is set, the request must fit
      within remaining = approvedFundingAmount
                       - totalDisbursedAmount
                       - sum(in-flight disbursements PENDING/APPROVED/PROCESSING)
                       - sum(PENDING StartupFundingRequests without disbursement).
  (d) Program/batch allocation ceiling: allocation headroom must be > 0.
      allocated = sum(allocations for (program, batch)) if any,
                  else sum(allocations for (program, batchId=null)) as fallback.
      used      = sum(disbursements for (program, batch) in status
                       PENDING | APPROVED | PROCESSING | COMPLETED).
      remaining = allocated - used.
      Disbursement amount must be <= remaining.

--------------------------------------------------------------------------------
3.1  POST /funding/program/:programId/disbursements - Create a disbursement
--------------------------------------------------------------------------------
Path params: programId (CUID).
Body:
  {
    "applicationId":    "cuid",           // REQUIRED; must be ONBOARDED
    "startupId":        "cuid",           // REQUIRED (service re-reads from app)
    "amount":           500000,           // > 0, REQUIRED
    "currency":         "INR",            // optional, default "INR"
    "disbursementType": "TRANCHE",        // LUMP_SUM | TRANCHE | MILESTONE_BASED
    "milestoneName":    "Prototype v1",   // optional
    "reference":        "UTR-123456",     // optional bank ref
    "notes":            "First tranche",  // optional
    "documents": [                        // optional; attached as DISBURSEMENT
      { "fileName": "receipt.pdf",
        "fileUrl":  "https://s3...",
        "fileType": "application/pdf",
        "description": "Bank advice" }
    ]
  }

Behavior:
  - For TRANCHE: trancheNumber = max(existing tranche# on this application)+1.
  - Created with status=PENDING.
  - disbursedById is set to the calling admin (audit only - the admin who
    recorded the transaction, not who marks it COMPLETED).

Errors (all 400 unless noted):
  - 404 "Application not found for this program"
  - 400 "Startup must be onboarded before funding can be disbursed"
  - 404 "Startup-Program association not found"
  - 400 "Disbursement exceeds remaining approved amount. Remaining: <x>, Requested: <y>"
  - 400 "No funding has been allocated to this program/batch yet. Allocate from a funding source first."
  - 400 "Disbursement exceeds allocation headroom. Remaining allocated: <x>, Requested: <y>"

Response data:
  FundingDisbursement with:
    - documents: FundingDocument[]
    - startup:   { id, name }

Audit:
  FundingHistory action=DISBURSEMENT_CREATED.

--------------------------------------------------------------------------------
3.2  GET /funding/program/:programId/disbursements - List for program
--------------------------------------------------------------------------------
Returns all disbursements for the program (all batches), newest first.
Each includes:
  - documents: FundingDocument[]
  - startup:   { id, name, logoUrl }
  - application: { id, status, requestedFundingAmount }

Frontend tip: group by `startupId` to render per-startup cards.

--------------------------------------------------------------------------------
3.3  GET /funding/disbursements/:disbursementId - Single disbursement
--------------------------------------------------------------------------------
Response data:
  FundingDisbursement with:
    - documents
    - startup:   { id, name, logoUrl }
    - application: { id, status, requestedFundingAmount, batchId }
    - program:   { id, title }
    - fundingRequest: { id, requestedAmount, status, note } | null
      (populated when this disbursement was auto-created from a
      StartupFundingRequest approval.)

Errors: 404 "Disbursement not found".

--------------------------------------------------------------------------------
3.4  PATCH /funding/disbursements/:disbursementId/status - Advance state
--------------------------------------------------------------------------------
Body:
  {
    "newStatus": "APPROVED" | "PROCESSING" | "COMPLETED" | "FAILED" | "CANCELLED",
    "reference": "UTR-789",        // optional; overwrites stored reference
    "notes":     "Approved"        // optional; overwrites stored notes
  }

Rules:
  - Must match an allowed transition (see state machine above); else
    400 "Cannot transition from <old> to <new>. Allowed: <list>".
  - Entering COMPLETED: sets disbursedAt=now() and
    association.totalDisbursedAmount += amount.
  - Leaving COMPLETED (COMPLETED -> CANCELLED): reverses
    association.totalDisbursedAmount -= amount.

Response data:
  Updated FundingDisbursement with the same includes as 3.1
  (documents, startup, application).

Audit (action depends on newStatus):
  APPROVED   -> DISBURSEMENT_APPROVED
  PROCESSING -> DISBURSEMENT_PROCESSING
  COMPLETED  -> DISBURSEMENT_COMPLETED
  FAILED     -> DISBURSEMENT_FAILED
  CANCELLED  -> DISBURSEMENT_CANCELLED

SECTION 4 - STARTUP-INITIATED REQUESTS (ADMIN REVIEW)
================================================================================
A StartupFundingRequest is created by a startup (see startup-api.txt). The
incubation admin reviews, approves (auto-creates a disbursement), or rejects.

--------------------------------------------------------------------------------
4.1  GET /funding/requests - List with filters
--------------------------------------------------------------------------------
Query params (all optional):
  programId, status, startupId, applicationId, batchId

Response data (array, newest first):
  StartupFundingRequest with:
    - startup:    { id, name, logoUrl }
    - program:    { id, title }
    - application:{ id, status, requestedFundingAmount }
    - disbursement: { id, status, amount, disbursedAt } | null
      (null unless status=APPROVED)

Frontend tip: default the inbox to `?status=PENDING`.

--------------------------------------------------------------------------------
4.2  GET /funding/requests/:requestId - Single request with full disbursement
--------------------------------------------------------------------------------
Response data:
  StartupFundingRequest with:
    - startup, program, application (same as above; application also
      includes batchId)
    - disbursement: full FundingDisbursement including `documents`
      (null when status != APPROVED).

Errors: 404 "Funding request not found".

--------------------------------------------------------------------------------
4.3  PATCH /funding/requests/:requestId/approve - Approve + auto-create disbursement
--------------------------------------------------------------------------------
Body:
  {
    "approvedAmount":   300000,      // optional; defaults to requestedAmount
    "disbursementType": "TRANCHE",   // optional; defaults to LUMP_SUM
    "currency":         "INR",       // optional
    "reference":        "UTR-789",   // optional
    "notes":            "Milestone 2 approved"  // optional
  }

Rules:
  - Request must have status=PENDING -> else 400.
  - approvedAmount (defaulting to request.requestedAmount) must pass:
      * association remaining (excluding THIS request from pending-sum), AND
      * allocation headroom.
  - For TRANCHE: auto-increments trancheNumber per application.

Side effects (single transaction):
  1. Creates FundingDisbursement (status=PENDING).
  2. Updates request: status=APPROVED, reviewedById, reviewNote=notes,
     approvedAmount, disbursementId.
  3. Writes BOTH audit rows: REQUEST_APPROVED and DISBURSEMENT_CREATED.

Response data:
  The updated StartupFundingRequest with `disbursement: FundingDisbursement`
  attached.

--------------------------------------------------------------------------------
4.4  PATCH /funding/requests/:requestId/reject
--------------------------------------------------------------------------------
Body:
  { "reviewNote": "Insufficient documentation" }  // optional string

Rules:
  - Request must have status=PENDING.
  - Sets status=REJECTED, reviewedById, reviewNote.

Response data:
  Updated StartupFundingRequest (not enriched).

Audit:
  FundingHistory action=REQUEST_REJECTED.

5.3  GET /funding/history - Audit trail
--------------------------------------------------------------------------------
Query params (all optional):

Frontend tip: on any detail page pass `?entityId=<id>` for per-entity

SECTION 6 - FRONTEND FLOW CHECKLISTS
================================================================================

Register a new grant and fund a cohort
  1. POST /funding/sources
  2. POST /funding/program/:programId/allocations (pass batchId)
  3. For each onboarded startup:
       POST /funding/program/:programId/disbursements
       PATCH /funding/disbursements/:id/status -> APPROVED
       PATCH /funding/disbursements/:id/status -> PROCESSING
       PATCH /funding/disbursements/:id/status -> COMPLETED

Review a startup request
  1. GET /funding/requests?status=PENDING[&programId=...]
  2. GET /funding/requests/:requestId for the drawer
  3. PATCH /funding/requests/:id/approve   (creates disbursement)
     OR
     PATCH /funding/requests/:id/reject
  4. For approved: advance the auto-created disbursement through the
     state machine via PATCH .../disbursements/:id/status.

Reverse a mistake after COMPLETED
  - PATCH /funding/disbursements/:id/status with newStatus=CANCELLED.
  - This decrements association.totalDisbursedAmount by the disbursement
    amount. It does NOT refund allocation headroom or source pool - those
    remain as they were at allocation time.

SECTION 7 - SCHEMA CHEAT SHEET (what the UI needs to render)
================================================================================

TenantFundingSource:
  id, tenantId, sourceName, sourceType, totalAmount, allocatedAmount,
  currency, receivedDate?, expiryDate?, reference?, notes?, isActive,
  createdAt, updatedAt, documents[], allocations[].

ProgramFundingAllocation:
  id, programId, fundingSourceId, batchId?, allocatedAmount, currency,
  allocatedById?, notes?, createdAt, updatedAt, fundingSource, program, batch?.

FundingDisbursement:
  id, programId, applicationId, startupId, tenantId, batchId?,
  amount, currency, disbursementType, status, trancheNumber?, milestoneName?,
  reference?, disbursedById?, disbursedAt?, notes?, createdAt, updatedAt,
  documents[], startup, application, program, fundingRequest?.

FundingDocument:
  id, ownerType (SOURCE | DISBURSEMENT), fundingSourceId?, disbursementId?,
  fileName, fileUrl, fileType?, description?, uploadedById?, createdAt.

StartupFundingRequest:
  id, applicationId, startupId, programId, tenantId, batchId?,
  requestedAmount, currency, note?, docUrl?, docName?, status,
  reviewedById?, reviewNote?, approvedAmount?, disbursementId?,
  createdAt, updatedAt, disbursement?, startup, program, application.

FundingHistory:
  id, tenantId, action, entityType, entityId, performedById?,
  oldValue?, newValue?, notes?, createdAt.

StartupProgramAssociation (read-only here):
  approvedFundingAmount?, totalDisbursedAmount (default 0), batchId?,
  status (ACTIVE|COMPLETED|OFFBOARDED|REMOVED).
