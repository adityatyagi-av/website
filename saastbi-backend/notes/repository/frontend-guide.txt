================================================================================
WHAT WE ARE BUILDING
================================================================================

The Public Repository feature allows incubation center admins to create custom
data repositories that are exposed through public API endpoints. Think of it
as a simplified, no-code API builder.

Real-world use cases:
  * A company directory that external websites can embed
  * An event schedule that partner organizations can consume
  * A startup portfolio listing for public display
  * A resource library with download links
  * A team member directory for public pages

The admin creates a repository, defines a custom schema (field names, types,
validation rules), adds data items, and gets a public API endpoint that anyone
can call without authentication.

================================================================================
PORTAL CONTEXT
================================================================================

This feature lives inside the INCUBATION PORTAL (admin dashboard).
  * Auth: Uses the portal authentication system (authenticatePortal middleware)
  * Multi-tenant: Each tenant (incubation center) manages their own repositories
  * Base URL: /api/incubation-portal

The incubation portal user (admin/staff) can:
  1. Create repositories with custom schemas
  2. Manage data items within repositories
  3. Monitor public API usage (stats + access logs)
  4. Regenerate API keys for security
  5. Activate/deactivate/archive repositories

================================================================================
DATA MODELS (WHAT THE FRONTEND WORKS WITH)
================================================================================

REPOSITORY OBJECT:
--------------------------------------------------------------------------------
{
  id: string (cuid)
  tenantId: string (cuid)
  name: string                    -- Display name (e.g., "Company Directory")
  slug: string                    -- URL-safe name (e.g., "company-directory")
  description: string | null      -- Optional description
  apiKey: string                  -- Public API key (e.g., "repo_a1b2c3...")
  status: "ACTIVE" | "INACTIVE" | "ARCHIVED"
  schema: SchemaField[]           -- Array of field definitions (see below)
  isRateLimited: boolean          -- Whether public API is rate limited
  rateLimitPerMinute: number      -- Max requests per minute per IP
  totalItems: number              -- Count of items in repository
  creatorId: string (cuid)        -- Who created it
  createdAt: string (ISO date)
  updatedAt: string (ISO date)
  _count: {
    items: number
    accessLogs: number
  }
}

SCHEMA FIELD OBJECT:
--------------------------------------------------------------------------------
{
  name: string                    -- Field identifier (camelCase, no spaces)
  type: RepositoryFieldType       -- One of the enum values below
  required: boolean               -- Whether the field is required
  description?: string            -- Help text shown to user
  maxLength?: number              -- For STRING/TEXT types
  min?: number                    -- For NUMBER type
  max?: number                    -- For NUMBER type
  options?: string[]              -- For SELECT/MULTI_SELECT types
}

RepositoryFieldType values:
  STRING        -> Short text input
  TEXT          -> Long text / textarea
  NUMBER        -> Numeric input
  BOOLEAN       -> Checkbox / toggle
  DATE          -> Date picker
  IMAGE_URL     -> URL input (for images)
  DOCUMENT_URL  -> URL input (for documents)
  URL           -> Generic URL input
  EMAIL         -> Email input
  PHONE         -> Phone number input
  SELECT        -> Dropdown / single select
  MULTI_SELECT  -> Multi-select / tag input

REPOSITORY ITEM OBJECT:
--------------------------------------------------------------------------------
{
  id: string (cuid)
  repositoryId: string (cuid)
  data: object                    -- Dynamic key-value pairs matching schema
  orderIndex: number              -- Sort order
  isActive: boolean               -- Whether visible in public API
  createdById: string (cuid)
  createdAt: string (ISO date)
  updatedAt: string (ISO date)
}

ACCESS LOG OBJECT:
--------------------------------------------------------------------------------
{
  id: string (cuid)
  repositoryId: string (cuid)
  ipAddress: string | null
  userAgent: string | null
  endpoint: string | null
  responseTime: number | null     -- In milliseconds
  createdAt: string (ISO date)
}

REPOSITORY STATS OBJECT:
--------------------------------------------------------------------------------
{
  totalItems: number
  activeItems: number
  access: {
    last24h: number
    last7d: number
    last30d: number
  }
  lastAccessedAt: string | null   -- ISO date
}

================================================================================
UI PAGES / VIEWS TO BUILD
================================================================================

The frontend needs these main views:

PAGE 1: REPOSITORY LIST PAGE
--------------------------------------------------------------------------------
Route suggestion: /portal/repositories

Purpose: Show all repositories for the tenant.

API: GET /api/incubation-portal/repositories

UI Elements:
  * Header: "Public Repositories" with "Create Repository" button
  * Filter bar:
    - Status dropdown: All | Active | Inactive | Archived
    - Search input: Searches name and description
  * Repository cards/table with columns:
    - Name + description
    - Status badge (green=ACTIVE, yellow=INACTIVE, gray=ARCHIVED)
    - Total items count
    - Total API accesses count (_count.accessLogs)
    - Created date
    - Action menu: View | Edit | Copy API Key | Archive
  * Pagination controls
  * Empty state: "No repositories yet. Create one to get started."

Key interactions:
  * Click card/row -> navigate to repository detail page
  * "Copy API Key" -> copy apiKey to clipboard with toast notification
  * "Create Repository" -> navigate to create page or open modal


PAGE 2: CREATE REPOSITORY PAGE
--------------------------------------------------------------------------------
Route suggestion: /portal/repositories/create

Purpose: Create a new repository with schema builder.

API: POST /api/incubation-portal/repository

UI Elements:
  * Form section 1 - Basic Info:
    - Name (text input, required)
    - Description (textarea, optional)

  * Form section 2 - Schema Builder (CRITICAL UI):
    - "Add Field" button
    - For each field, a row/card with:
      - Field Name (text input, alphanumeric + underscore, required)
      - Field Type (dropdown of RepositoryFieldType values)
      - Required toggle (checkbox)
      - Description (text input, optional)
      - Type-specific options:
        * STRING/TEXT: maxLength (number input)
        * NUMBER: min + max (number inputs)
        * SELECT/MULTI_SELECT: options builder (add/remove string tags)
    - Reorder fields via drag-and-drop
    - Remove field button on each row

  * Form section 3 - Rate Limiting:
    - Enable rate limiting (toggle, default: on)
    - Requests per minute (number input, default: 60)

  * Submit button: "Create Repository"

After creation:
  * Show success state with the generated API key
  * Highlight that the API key should be saved/shared
  * Show the public API URL format
  * Option to "Add Items Now" or "Go to Repository"

Schema Builder UX Tips:
  * Show visual previews of what each field type looks like
  * Validate field names in real-time (must match /^[a-zA-Z][a-zA-Z0-9_]*$/)
  * Prevent duplicate field names
  * For SELECT/MULTI_SELECT, provide a tag-style input for options


PAGE 3: REPOSITORY DETAIL PAGE
--------------------------------------------------------------------------------
Route suggestion: /portal/repositories/:repositoryId

Purpose: View and manage a single repository.

APIs used:
  * GET /api/incubation-portal/repository/:repositoryId
  * GET /api/incubation-portal/repository/:repositoryId/stats
  * GET /api/incubation-portal/repository/:repositoryId/items
  * GET /api/incubation-portal/repository/:repositoryId/access-logs

UI Sections:

  Section A - Header:
    - Repository name + status badge
    - Edit button (opens edit modal/page)
    - Status toggle (ACTIVE <-> INACTIVE)
    - Archive button
    - "Regenerate API Key" button (with confirmation dialog!)

  Section B - API Information Card:
    - Public API URL: {BASE_URL}/api/public/repository/{apiKey}
    - Schema URL: {BASE_URL}/api/public/repository/{apiKey}/schema
    - API Key display (masked by default, click to reveal)
    - "Copy URL" and "Copy API Key" buttons
    - Rate limit info: "{rateLimitPerMinute} requests/min" or "Unlimited"

  Section C - Statistics Dashboard:
    - Total items / Active items
    - API calls last 24h / 7d / 30d (use stat cards)
    - Last accessed timestamp
    - Optional: Simple bar chart of daily access over last 7 or 30 days

  Section D - Schema Preview:
    - Table/list of field definitions
    - Columns: Name | Type | Required | Constraints | Description
    - Visual type icons for each field type

  Section E - Data Items Tab:
    - Table showing all items
    - Columns generated dynamically from schema fields
    - Each column header = field name from schema
    - "Add Item" button -> opens add item modal/drawer
    - "Bulk Add" button -> opens bulk add modal (paste JSON or form)
    - Inline actions per row: Edit | Delete
    - Bulk select + "Delete Selected" for bulk delete
    - Pagination

  Section F - Access Logs Tab:
    - Table of access logs
    - Columns: IP Address | User Agent | Endpoint | Response Time | Date
    - Pagination
    - Optional: Filter by date range


PAGE 4: ADD/EDIT ITEM MODAL OR DRAWER
--------------------------------------------------------------------------------
Purpose: Add or edit a single data item.

APIs:
  * POST /api/incubation-portal/repository/:repositoryId/item
  * PUT /api/incubation-portal/repository/:repositoryId/item/:itemId

UI: Dynamic form generated from the repository schema.

For each field in schema, render the appropriate input:
  * STRING     -> <input type="text" maxLength={field.maxLength}>
  * TEXT       -> <textarea maxLength={field.maxLength}>
  * NUMBER     -> <input type="number" min={field.min} max={field.max}>
  * BOOLEAN    -> <input type="checkbox"> or toggle switch
  * DATE       -> <input type="date"> or date picker component
  * IMAGE_URL  -> <input type="url"> with image preview
  * DOCUMENT_URL -> <input type="url">
  * URL        -> <input type="url">
  * EMAIL      -> <input type="email">
  * PHONE      -> <input type="tel">
  * SELECT     -> <select> with field.options
  * MULTI_SELECT -> Multi-select component / checkboxes with field.options

Mark required fields with asterisk (*).
Show field description as help text below each input.
Show validation errors from the API (400 response lists all field errors).

When editing: pre-populate form with item.data values.


PAGE 5: EDIT REPOSITORY PAGE/MODAL
--------------------------------------------------------------------------------
Purpose: Edit repository settings and schema.

API: PUT /api/incubation-portal/repository/:repositoryId

UI: Similar to create page but pre-populated.
  * Name and description editable
  * Schema builder pre-populated with current fields
  * Rate limit settings editable
  * WARNING: Show a notice that changing schema does NOT re-validate existing items

================================================================================
API CALLS REFERENCE (FOR FRONTEND)
================================================================================

All portal APIs require:
  Headers:
    Authorization: Bearer <token>
    tenantkey: <tenantKey>
    Content-Type: application/json

1. LIST REPOSITORIES
   GET /api/incubation-portal/repositories?page=1&limit=20&status=ACTIVE&search=...

2. CREATE REPOSITORY
   POST /api/incubation-portal/repository
   Body: { name, description, schema, isRateLimited, rateLimitPerMinute }

3. GET REPOSITORY
   GET /api/incubation-portal/repository/:repositoryId

4. UPDATE REPOSITORY
   PUT /api/incubation-portal/repository/:repositoryId
   Body: { name?, description?, status?, schema?, isRateLimited?, rateLimitPerMinute? }

5. DELETE (ARCHIVE) REPOSITORY
   DELETE /api/incubation-portal/repository/:repositoryId

6. REGENERATE API KEY
   POST /api/incubation-portal/repository/:repositoryId/regenerate-key

7. GET STATS
   GET /api/incubation-portal/repository/:repositoryId/stats

8. GET ACCESS LOGS
   GET /api/incubation-portal/repository/:repositoryId/access-logs?page=1&limit=50

9. ADD ITEM
   POST /api/incubation-portal/repository/:repositoryId/item
   Body: { [fieldName]: value, ... }

10. LIST ITEMS
    GET /api/incubation-portal/repository/:repositoryId/items?page=1&limit=20&isActive=true

11. GET ITEM
    GET /api/incubation-portal/repository/:repositoryId/item/:itemId

12. UPDATE ITEM
    PUT /api/incubation-portal/repository/:repositoryId/item/:itemId
    Body: { [fieldName]: value, ... }

13. DELETE ITEM
    DELETE /api/incubation-portal/repository/:repositoryId/item/:itemId

14. BULK ADD ITEMS
    POST /api/incubation-portal/repository/:repositoryId/items/bulk
    Body: { items: [{ [fieldName]: value }, ...] }

15. BULK DELETE ITEMS
    DELETE /api/incubation-portal/repository/:repositoryId/items/bulk
    Body: { itemIds: ["cuid_1", "cuid_2", ...] }

IMPORTANT FRONTEND CONSIDERATIONS
================================================================================

1. API KEY SECURITY
   - Never expose API keys in URLs visible to unauthorized users
   - Mask API keys by default (show first 10 chars + "...")
   - Add "click to reveal" and "copy to clipboard" interactions
   - Regenerate key button should have a confirmation dialog:
     "This will invalidate the current API key immediately.
      All external consumers will need the new key."

2. DYNAMIC FORM GENERATION
   - The schema defines the form structure dynamically
   - Map each RepositoryFieldType to the correct HTML input/component
   - Handle validation client-side first, then rely on server validation
   - Display all server validation errors (the API returns them as semicolon-joined string)

3. STATUS MANAGEMENT
   - ACTIVE: Green badge. Repository serving data.
   - INACTIVE: Yellow/amber badge. Paused, not serving.
   - ARCHIVED: Gray badge. Soft deleted.
   - Provide easy toggle between ACTIVE and INACTIVE
   - Archive should have confirmation dialog

4. SCHEMA BUILDER
   - This is the most complex UI component
   - Allow adding, removing, reordering fields
   - Each field type has different constraint options
   - Show/hide constraint inputs based on selected type:
     * STRING/TEXT: show maxLength
     * NUMBER: show min, max
     * SELECT/MULTI_SELECT: show options builder
     * Others: no extra constraints
   - Validate field names (alphanumeric + underscore, starts with letter)
   - Prevent duplicate field names

5. ITEMS TABLE (DYNAMIC COLUMNS)
   - Column headers come from schema field names
   - Render cell values based on field type:
     * BOOLEAN: checkmark/cross icon
     * URL/IMAGE_URL/DOCUMENT_URL: clickable link
     * EMAIL: mailto link
     * PHONE: tel link
     * DATE: formatted date string
     * SELECT: badge/chip
     * MULTI_SELECT: multiple badges/chips
     * STRING/TEXT/NUMBER: plain text
   - Long text values should truncate in table with tooltip
   - Consider limiting visible columns for wide schemas

6. PUBLIC API URL DISPLAY
   - Show the full public URL prominently on the detail page
   - Format: {window.location.origin}/api/public/repository/{apiKey}
   - Provide copy button
   - Optionally show a code snippet for how to fetch data:
     ```
     fetch('https://your-domain.com/api/public/repository/repo_abc123')
       .then(res => res.json())
       .then(data => console.log(data));
     ```

7. BULK OPERATIONS
   - Bulk add: Accept JSON array input or provide multi-row form
   - Bulk delete: Checkbox selection on items table + "Delete Selected" button
   - Show progress/status for bulk operations
   - Display individual item errors from bulk add (the API specifies which index failed)

8. ACCESS LOG ANALYTICS
   - Show basic stats prominently (24h, 7d, 30d counters)
   - Access log table with IP, user agent, response time
   - Consider grouping by IP for overview
   - Timestamps should be in user's local timezone

9. ERROR HANDLING
   - 400 errors from item add/update contain detailed field validation messages
   - Parse the semicolon-separated error string into individual field errors
   - Display errors next to the relevant form fields when possible
   - Handle 404 (repository not found) gracefully
   - Handle 429 (rate limit) with appropriate user message

10. EMPTY STATES
    - No repositories: "Create your first public repository"
    - No items in repository: "Add data to your repository"
    - No access logs: "No one has accessed this repository's API yet"
