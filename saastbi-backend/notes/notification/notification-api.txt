================================================================================
NOTIFICATION MODULE — FRONTEND API DOCUMENTATION
================================================================================

Version: 1.1.0
Last Updated: 2026-03-07

This module is available across THREE portals with identical endpoints:
  - Ecosystem:         Base URL: /api/ecosystem
  - Incubation Portal: Base URL: /api/incubation-portal
  - Startup Portal:    Base URL: /api/startup-portal

Authentication:
  - Ecosystem:         User Bearer Token (authenticate middleware)
  - Incubation Portal: Portal Bearer Token (authenticatePortal middleware)
  - Startup Portal:    User Bearer Token (authenticate middleware)

Note: All three portals share the same notification storage. The incubation
portal resolves the portal user's linked ecosystem userId internally.
Startup portal users authenticate as ecosystem Users directly (req.user.id),
so no resolution is needed.

================================================================================
ARCHITECTURE OVERVIEW
================================================================================

The notification system has three delivery channels:

1. REST API    — Fetch, list, paginate, manage, and configure notifications
2. Socket.IO   — Real-time delivery via the /notifications namespace
3. Email       — Automatic background dispatch for supported notification types

Key Concepts:
- Every notification belongs to a single user (userId = ecosystem User.id)
- Notifications have a TYPE (what happened) and CATEGORY (which module)
- Users can configure per-category channel preferences (in-app, email, push)
- Grouped notifications merge similar rapid-fire events (e.g., "3 people liked")
- Notifications can be archived (soft-hide) then permanently purged by cron

DATA MODELS
================================================================================

Notification:
  - id: string (cuid)
  - userId: string (ecosystem User.id, always)
  - type: NotificationType (see enum below)
  - category: NotificationCategory (default: "SYSTEM")
  - priority: NotificationPriority (default: "MEDIUM")
  - title: string
  - message: string
  - data: JSON | null (arbitrary payload, e.g., { actors: [...], count: 3 })
  - isRead: boolean (default: false)
  - readAt: DateTime | null
  - actionUrl: string | null (deep link for frontend navigation)
  - actorId: string | null (who triggered the notification)
  - actorName: string | null
  - actorAvatar: string | null
  - entityType: string | null (e.g., "MentorSession", "CommunityPost", "Post")
  - entityId: string | null (ID of the related entity)
  - groupKey: string | null (for grouped notifications, e.g., "post:123:like")
  - isArchived: boolean (default: false)
  - expiresAt: DateTime | null (auto-deleted by cron when expired)
  - createdAt: DateTime
  - updatedAt: DateTime

NotificationPreference:
  - id: string (cuid)
  - userId: string
  - category: NotificationCategory
  - inApp: boolean (default: true)
  - email: boolean (default: true)
  - push: boolean (default: false)
  - createdAt: DateTime
  - updatedAt: DateTime
  - Unique constraint: [userId, category]

6. ARCHIVE NOTIFICATION
================================================================================

Method: DELETE
URL: {BASE}/notification/:id
Auth: Token Required

Purpose: Archive a notification (soft-delete). Archived notifications are
hidden from listings but not permanently deleted. The cron job permanently
purges archived notifications older than 90 days.

Path Parameters:
- id (string, REQUIRED) — Notification ID

Response (200):
{
  "success": true,
  "message": "Notification archived",
  "data": {
    "id": "cuid_notif_1",
    "isArchived": true,
    ... (all fields)
  }
}

Response (404):
{
  "success": false,
  "message": "Notification not found"
}

Frontend Notes:
- Use for swipe-to-dismiss or delete button on notification items
- Archived notifications don't appear in GET /notification listings
- Unread count updated via socket after archiving

SOCKET.IO — REAL-TIME NOTIFICATIONS
================================================================================

Namespace: /notifications
Transport: WebSocket with polling fallback

--------------------------------------------------------------------------------
CONNECTION & AUTHENTICATION
--------------------------------------------------------------------------------

Connect to the /notifications namespace:

  const socket = io("{{SERVER_URL}}/notifications", {
    auth: { token: "Bearer <jwt_token>" }
  });

The token can be either:
  - An ecosystem User JWT
  - An incubation portal JWT (userId is resolved internally via linked account)

On successful connection:
  - User auto-joins room "notifications:{userId}"
  - Server immediately emits "notification:unread-count" with current counts

On auth failure:
  socket.on("connect_error", (err) => {
    console.error(err.message);
    // "Authentication error: No token provided"
    // "Authentication error: User not found or not linked"
    // "Authentication error: jwt expired"
  });

================================================================================
NOTIFICATION BEHAVIOR RULES
================================================================================

1. SELF-NOTIFICATION SUPPRESSION
   - If actorId === recipientId, the notification is silently dropped
   - Example: liking your own post does not generate a notification

2. BLOCK RESPECT
   - If the recipient has blocked the actor (UserBlock), notification is dropped
   - This uses the UserBlock table's blockerId_blockedId compound unique

3. CHANNEL PREFERENCES
   - inApp: false for a category = notification is not created at all
   - email: false = notification created, but no email sent
   - email is further gated by UserSettings.emailNotifications global toggle
   - push: reserved for future use

4. GROUPED NOTIFICATIONS
   - Notifications with the same groupKey within 15 minutes are merged
   - Only unread, non-archived notifications are eligible for grouping
   - The message is rewritten: "Alice and 2 others liked your post"
   - data.actors array tracks all contributing actors
   - data.count tracks the total count
   - Frontend receives "notification:updated" for merged notifications

5. AUTOMATIC LIFECYCLE
   - Notifications with expiresAt set are deleted by cron when expired (3 AM)
   - Read notifications older than 30 days are auto-archived (3:30 AM)
   - Archived notifications older than 90 days are permanently deleted (4 AM)
   - Session reminders are sent 1 hour before (every 15 min cron)

================================================================================
COMPLETE FRONTEND FLOWS
================================================================================

Flow 1: Notification Bell / Dropdown
  1. Connect to Socket.IO /notifications namespace on app load
  2. Receive "notification:unread-count" on connection — set badge
  3. Listen for "notification:new" — increment badge, show toast
  4. Listen for "notification:updated" — update existing item
  5. On bell click: GET /notification?page=1&limit=20
  6. Render list with title, message, actorAvatar, time ago
  7. Unread items get visual accent (bold/dot)
  8. On notification click:
     a. PATCH /notification/:id/read (or socket "notification:mark-read")
     b. Navigate to actionUrl
  9. "Mark all as read" button: PATCH /notification/read-all

Flow 2: Full Notification Page with Category Tabs
  1. GET /notification/unread-count — build tab badges
  2. GET /notification?page=1&limit=20 — "All" tab
  3. On tab change: GET /notification?category=MENTORSHIP&page=1&limit=20
  4. Per-tab "Mark all as read": PATCH /notification/read-all { category }
  5. Swipe/dismiss: DELETE /notification/:id (archive)
  6. Infinite scroll: increment page param

Flow 3: Notification Preferences Settings
  1. GET /notification/preferences
  2. Render toggles for each category: In-App | Email | Push
  3. On toggle change: PUT /notification/preferences { preferences: [...] }
  4. Show success toast on update

Flow 4: Grouped Notification Display
  1. Check if data.actors exists and data.count > 1
  2. Show avatar stack (first 3 actors from data.actors)
  3. Display aggregated message (e.g., "Alice and 2 others liked your post")
  4. On click: navigate to actionUrl (the shared entity)

Flow 5: Cleanup
  1. Swipe-to-dismiss archives individual notifications
  2. "Clear all dismissed" button: DELETE /notification/clear
  3. Automatic cleanup happens via server cron — no frontend action needed

================================================================================
MULTI-PORTAL IDENTITY RESOLUTION
================================================================================

All three portals (Ecosystem, Incubation, Startup) write to the same
notification table using the ecosystem User.id as the userId.

For Ecosystem users:
  - req.user.id IS the User.id directly

For Incubation portal users:
  - req.user.id is the IncubationUser.id
  - The controller resolves this to IncubationUser.userId (linked User.id)
  - If IncubationUser.userId is null (not linked), returns 403

For Startup portal users:
  - req.user.id IS the User.id directly (authenticate middleware)
  - Uses StartupNotificationController (identical logic to ecosystem)
  - No userId resolution needed — startup members authenticate as ecosystem Users

This means:
  - A user logged into any portal sees the SAME notifications
  - Notifications sent from the mentor module (which uses ecosystem User.id)
    appear in all portals
  - The Socket.IO namespace also resolves both token types to the same userId

Pattern Key:
  send()        — Single recipient notification
  sendBulk()    — Same notification to multiple recipients
  sendGrouped() — Grouped/aggregated notification (merges within 15-min window)
  All triggers use fire-and-forget: .catch(() => {})

--- Social (user-social.service.js) ---
  FOLLOW                 | sendGrouped | "started following you"
  CONNECTION_REQUEST     | send        | "{name} wants to connect with you"
  CONNECTION_ACCEPTED    | send        | "{name} accepted your connection request"

--- Events (event-registration.service.js) ---
  EVENT_REGISTRATION_CONFIRMED | sendBulk | "Your registration has been confirmed"
  EVENT_WAITLIST_PROMOTED      | sendBulk | "You've been promoted from the waitlist"

--- Jobs (job-application.service.js) ---
  JOB_NEW_APPLICATION    | send    | "{name} applied for {job title}"

--- Tasks (task.service.js) ---
  TASK_ASSIGNED          | send    | "{name} assigned you the task {title}"
  TASK_STATUS_CHANGED    | send    | "{name} changed task {title} status to {status}"

--- Mentorship (mentorship.service.js) ---
  MENTORSHIP_REQUEST     | send    | "You have received a new mentorship request"
  MENTORSHIP_ACCEPTED    | send    | "Your mentorship request has been accepted"

--- Sessions (session.service.js) ---
  SESSION_BOOKED         | send    | "You have a new session booking"
  SESSION_CONFIRMED      | send    | "Your mentoring session has been confirmed"
  SESSION_DECLINED       | send    | "Your session request was declined: {reason}"
  SESSION_NO_SHOW        | send    | "The mentor marked you as a no-show"
  SESSION_REMINDER       | send    | "Your session starts in about 1 hour" (cron)

--- Incubator (incubator-association.service.js) ---
  INCUBATOR_INVITATION           | send | "Invitation from an incubator"
  INCUBATOR_APPLICATION_APPROVED | send | "Your application has been approved"
  INCUBATOR_APPLICATION_REJECTED | send | "Your application was rejected"

--- Community (community-notification.service.js) ---
  COMMUNITY_POST         | sendBulk | "A new post was shared in your community"
  COMMUNITY_MENTION      | sendBulk | "You were mentioned in {community}"
  COMMUNITY_COMMENT      | send     | Comment preview (max 100 chars)
  COMMUNITY_JOIN_REQUEST | sendBulk | "{name} wants to join {community}"
  COMMUNITY_JOIN_APPROVED| send     | "Your request to join {community} approved"
  COMMUNITY_INVITE       | send     | "{name} invited you to join {community}"
  SYSTEM (moderation)    | sendBulk | "A {type} has been reported in {community}"

--- Community Engagement (community-engagement.service.js) ---
  COMMUNITY_POST (like)    | sendGrouped | "liked your community post"
  COMMUNITY_COMMENT        | sendGrouped | "commented on your community post"
