4. MATCH SCORE COMPUTATION FLOW
================================================================================

Input:
  - User Profile: skills, experiences, educations, location
  - Job: requiredSkills, niceToHaveSkills, experienceLevel, workMode, location, educationLevel

Computation:
--------------------------------------------------------------------------------

┌──────────────────────────────────────────────────────────────────────────┐
│                     MATCH SCORE ALGORITHM (0-100)                        │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  1. REQUIRED SKILLS (max 40 points)                                      │
│     matched = requiredSkills ∩ userSkills                                │
│     score = (matched.length / requiredSkills.length) * 40                │
│     If no required skills defined: default 20                            │
│                                                                          │
│  2. NICE-TO-HAVE SKILLS (max 10 points)                                  │
│     matched = niceToHaveSkills ∩ userSkills                              │
│     score = (matched.length / niceToHaveSkills.length) * 10              │
│                                                                          │
│  3. EXPERIENCE LEVEL (max 20 points)                                     │
│     Calculate user level from total years:                               │
│       0-1yr = ENTRY, 1-2yr = JUNIOR, 2-4yr = MID                        │
│       4-7yr = SENIOR, 7-10yr = LEAD, 10+yr = EXECUTIVE                  │
│     Exact match = 20, 1 level off = 12, 2 levels = 5, 3+ = 0            │
│                                                                          │
│  4. WORK MODE / LOCATION (max 10 points)                                 │
│     Remote job = 10                                                      │
│     City/country match = 10                                              │
│     Location unknown = 5                                                 │
│     No match = 2                                                         │
│                                                                          │
│  5. EDUCATION (max 10 points)                                            │
│     Has education + job requires it = 10                                 │
│     Has education, no requirement = 7                                    │
│     No education data = 3                                                │
│                                                                          │
│  TOTAL = min(sum of all, 100)                                            │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘

Example:
  User: React (5yr), Node.js (4yr), TypeScript (3yr), Senior level, SF
  Job: Required [React, Node.js, PostgreSQL], Senior, Hybrid, SF

  Skills:    2/3 matched = 26.7 → 27 points
  Bonus:     0 (no nice-to-have skills defined)
  Experience: SENIOR == SENIOR = 20 points
  Location:  SF matches SF = 10 points
  Education: Has degree = 10 points
  TOTAL: 67/100

5. SCREENING EVALUATION FLOW
================================================================================

When an applicant submits screening answers:

1. Application is created with screening answers
2. evaluateScreeningAnswers() is called
3. Only ELIMINATORY questions with expectedAnswer are scored
4. Score = (correct / total eliminatory) * 100
5. screeningScore is saved to the application

Example:
  Q1: "Years of React experience?" (NUMERIC, eliminatory, expected: 3)
      Answer: 5 → Match (5 >= 3 is compared via JSON stringify, not numeric)
  Q2: "Authorized to work in US?" (YES_NO, eliminatory, expected: true)
      Answer: true → Match
  Q3: "Why interested?" (TEXT, not eliminatory)
      Answer: "..." → Not scored

  Score: 2/2 = 100%

9. ROUTE ORDERING RATIONALE
================================================================================

Routes are ordered carefully to avoid Express param conflicts:

1. Static routes FIRST:
   /job/discover, /job/recommended, /job/trending, /job/bookmarked
   /job/my-applications, /job/my-referrals, /job/alerts

2. Named param routes with prefixes:
   /job/by-skill/:skillName
   /job/page/:pageId
   /job/alert/:alertId
   /job/application/:applicationId
   /job/screening-question/:questionId
   /job/applicant/:applicantId

3. Dynamic :jobId routes:
   /job/:jobId/apply, /job/:jobId/status, etc.

4. Dynamic :slug route LAST:
   /job/:slug (catches everything not matched above)

This prevents Express from treating "discover" or "recommended"
as a :slug or :jobId parameter.
