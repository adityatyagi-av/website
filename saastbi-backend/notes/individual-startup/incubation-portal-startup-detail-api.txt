================================================================================
INDIVIDUAL STARTUP DETAIL APIs - INCUBATION PORTAL (FRONTEND REFERENCE)
================================================================================

Audience: frontend engineers (or an LLM generating UI code) building the
incubation-portal individual startup detail screens. Every endpoint in this
document is derived from:

  src/routes/incubation/portal/startup.route.js
  src/controllers/incubation/portal/startup-detail.controller.js
  src/services/incubation/portal/startup-detail.service.js
  prisma/schema.prisma (models Startup, StartupApplication, Evaluation,
                       FundingDisbursement, StartupFundingRequest,
                       StartupProgramAssociation, OfficeAllocation,
                       OfficeBooking, OfficePayment, FacilityBooking,
                       Mentorship, MentorSession, SchemeAnswer, etc.)

Global Conventions
------------------
- Base URL prefix: /api/incubation-portal
- Auth header:     Authorization: Bearer <portal token>     (authenticatePortal)
- Access gate:     requireAccessByMethod(MODULE_KEYS.STARTUP_MANAGEMENT)
                   All routes below are gated on the STARTUP_MANAGEMENT module.
- Tenant context:  tenantId is resolved from the authenticated user's token
                   (req.user.tenantId). No separate tenantkey header needed.
- Response envelope (apiResponse.sendSuccess):
    {
      "success": true,
      "message": "<human message>",
      "data":    <payload described per endpoint>
    }
- Errors:
    { "success": false, "message": "<reason>", "statusCode": 4xx }
- All IDs are CUIDs. All timestamps are ISO-8601 strings.
- All endpoints verify the startup belongs to the tenant via
  StartupTenantAssociation before returning data.

Who can access these APIs:
  - Incubation admin
  - Program managers
  Any user with STARTUP_MANAGEMENT module access on the incubation portal.

--------------------------------------------------------------------------------
2.2  POST /startups/:startupId/funding/disburse - Disburse funds to startup
--------------------------------------------------------------------------------
Path params:
  - startupId (string, REQUIRED)

Body (application/json):
  {
    "applicationId":    "clx...",        // REQUIRED - StartupApplication ID
    "programId":        "clx...",        // REQUIRED - Program the disbursement is for
    "batchId":          "clx...",        // optional - Batch ID (auto-detected from application if omitted)
    "amount":           500000,          // REQUIRED - float > 0
    "currency":         "INR",           // optional, default "INR"
    "disbursementType": "TRANCHE",       // REQUIRED - LUMP_SUM | TRANCHE | MILESTONE_BASED
    "milestoneName":    "MVP Completion",// optional - relevant for MILESTONE_BASED
    "reference":        "TXN-002",       // optional - external reference
    "notes":            "Second tranche",// optional
    "documents": [                       // optional - supporting documents
      {
        "fileName": "sanction_letter.pdf",
        "fileUrl":  "https://s3...",
        "fileType": "application/pdf",
        "description": "Sanction letter"
      }
    ]
  }

Validation rules:
  - Startup must be ONBOARDED in the program (application status = "ONBOARDED")
  - Amount must not exceed remaining approved amount for the startup-program
    association (approvedFundingAmount - totalDisbursed - in-flight)
  - Amount must not exceed allocation headroom for program/batch
  - For TRANCHE type, trancheNumber is auto-incremented

Response (201):
  Same shape as a single object from disbursements array above, plus:
  "startup": { "id": "clx...", "name": "TechVenture" },
  "program": { "id": "clx...", "title": "Accelerator 2026" }

Errors:
  400 - "applicationId, programId, amount, and disbursementType are required"
  400 - "Startup must be onboarded before funding can be disbursed"
  400 - "Disbursement exceeds remaining approved amount..."
  400 - "No funding has been allocated to this program/batch yet"
  400 - "Disbursement exceeds allocation headroom..."
  404 - "Startup not associated with this tenant"
  404 - "Application not found for this startup/program"
  404 - "Startup-Program association not found"

SECTION 10 - FRONTEND INTEGRATION NOTES
================================================================================

1. Suggested page structure:
   - Tab 1: Overview (use /overview endpoint)
   - Tab 2: Funding (use /funding endpoint, disburse action)
   - Tab 3: Registration (use /registration/:applicationId, pick from apps list)
   - Tab 4: Evaluations (use /evaluations endpoint)
   - Tab 5: Office & Facilities (use /office-allocations and /facility-bookings)
   - Tab 6: Mentorship (use /mentorships endpoint)
   - Tab 7: Associations (use /associations endpoint)

2. The overview tab gives enough info to show cards/widgets with links to
   detailed tabs. Load the overview first, then lazy-load other tabs.

3. For the registration tab, first fetch basic startup details from
   GET /startups/get-startup/:startupId to get the applications array,
   then let the user pick which application's registration to view.

4. For funding disbursement, the frontend needs to know:
   - Which applications are ONBOARDED (from /funding -> programFunding)
   - The remaining budget per association (approvedFundingAmount - totalDisbursed)
   - Show a form with applicationId, programId pre-filled

5. Evaluation scores are grouped by application. If user wants per-program
   view, use the applicationId query filter.

6. Office allocations returns 3 separate arrays:
   - allocations (long-term space assignments)
   - bookings (individual room bookings)
   - payments (payment records for all office activities)

7. Facility bookings are paginated. Use fromDate/toDate for calendar views.

8. For the associations view, note that it shows ALL incubators and programs
   the startup is part of (not just the current tenant). This is intentional
   to give a complete picture of the startup's ecosystem involvement.
