1. OVERVIEW
================================================================================

The Event module handles the full lifecycle of events: creation, publishing,
registration, ticketing, payments, live streaming, Q&A, polls, and feedback.

All event APIs live under ONE portal (Ecosystem). There is no "two portal"
split like the Office module. The same user can be a HOST or an ATTENDEE.

WHO CAN DO WHAT:
  HOST = The person who created the event, OR an OWNER/ADMIN of the Page
         that hosts the event.
  ATTENDEE = Any logged-in user on the platform.
  ANYONE = Works without login too (public browsing).

================================================================================
2. AUTHENTICATION & HEADERS
================================================================================

There are TWO types of endpoints:

  1. AUTHENTICATED (requires login)
     Header: Authorization: Bearer <access_token>

  2. OPTIONAL AUTH (works without login, but adds extra data if logged in)
     Header: Authorization: Bearer <access_token>  (optional)
     When logged in: response includes isBookmarked, registrationStatus, etc.
     When not logged in: those fields are not present.

Every endpoint below is marked with:
  Auth: Required      → must be logged in
  Auth: Optional      → works either way

================================================================================
3. STANDARD RESPONSE FORMAT
================================================================================

Every API response follows this structure:

SUCCESS:
{
  "success": true,
  "status": 200,
  "data": { ... },
  "message": "Event created successfully"
}

ERROR:
{
  "success": false,
  "status": 400,
  "data": null,
  "message": "Event is not accepting registrations"
}

Status codes used:
  200 → Success (GET, PATCH, DELETE)
  201 → Created (POST that creates something)
  400 → Bad request (invalid input, wrong state)
  401 → Not logged in
  403 → Not authorized (not the host)
  404 → Not found
  409 → Conflict (already exists, already registered)

PAGINATION (when present):
{
  "data": [ ... ],
  "pagination": {
    "total": 50,
    "page": 1,
    "limit": 10,
    "totalPages": 5
  }
}

================================================================================
4. ALLOWED VALUES (ENUMS)
================================================================================

Use these exact string values in request bodies and query parameters.

EVENT TYPE (eventType):
  CONFERENCE, WORKSHOP, MEETUP, WEBINAR, PITCH_EVENT, HACKATHON,
  NETWORKING, DEMO_DAY, PANEL_DISCUSSION, BOOTCAMP, SEMINAR, OTHER

EVENT FORMAT (format):
  IN_PERSON, VIRTUAL, HYBRID

EVENT STATUS (returned, not sent):
  DRAFT, PUBLISHED, CANCELLED, COMPLETED

REGISTRATION STATUS (returned, not sent):
  REGISTERED, CONFIRMED, WAITLISTED, CANCELLED, ATTENDED, NO_SHOW

PAYMENT STATUS (returned, not sent):
  PENDING, COMPLETED, FAILED, REFUNDED

LIVE STATUS (returned, not sent):
  NOT_STARTED, LIVE, PAUSED, ENDED

RECURRING TYPE (recurringType):
  NONE, DAILY, WEEKLY, BIWEEKLY, MONTHLY

SPONSOR TIER (tier):
  PLATINUM, GOLD, SILVER, BRONZE, PARTNER

MEDIA TYPE (mediaType):
  IMAGE, VIDEO, DOCUMENT

INVITATION STATUS (returned, not sent):
  PENDING, ACCEPTED, DECLINED

ORGANIZER ROLE (role):
  host, co_host, moderator, speaker, organizer

5. FLOW 1: CREATE & PUBLISH EVENT
================================================================================

This flow is performed by the event host (any logged-in user or a Page admin).

Step 1: Create Event
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event
Auth: Required

Body:
{
  "title": "Tech Meetup 2026",                   ← required
  "description": "A meetup for tech enthusiasts", ← required
  "shortDesc": "Quick summary",                   ← optional
  "coverImage": "https://s3.../image.jpg",        ← optional (S3 URL)
  "eventType": "MEETUP",                          ← required (see enums)
  "format": "IN_PERSON",                          ← optional, default: IN_PERSON
  "pageId": "page-uuid",                          ← optional (host as a Page)
  "startDate": "2026-03-15T10:00:00Z",            ← required
  "endDate": "2026-03-15T18:00:00Z",              ← required
  "timezone": "Asia/Kolkata",                     ← optional, default: Asia/Kolkata
  "venue": "Convention Center",                   ← optional
  "address": "123 Main St",                       ← optional
  "city": "Bangalore",                            ← optional
  "state": "Karnataka",                           ← optional
  "country": "India",                             ← optional
  "latitude": 12.9716,                            ← optional
  "longitude": 77.5946,                           ← optional
  "virtualUrl": "https://...",                    ← optional
  "capacity": 100,                                ← optional (null = unlimited)
  "isPaid": false,                                ← optional, default: false
  "price": 500,                                   ← optional
  "currency": "INR",                              ← optional, default: INR
  "tags": ["tech", "networking"],                 ← optional (string array)
  "isPublic": true,                               ← optional, default: true
  "requiresApproval": false,                      ← optional, default: false
  "allowWaitlist": false,                         ← optional, default: false
  "autoConfirm": true,                            ← optional, default: true
  "registrationDeadline": "2026-03-14T23:59:00Z", ← optional
  "maxLiveParticipants": 500,                     ← optional
  "isRecurring": false,                           ← optional, default: false
  "recurringType": "WEEKLY",                      ← required if isRecurring=true
  "recurringDays": ["MONDAY", "WEDNESDAY"],       ← optional (for WEEKLY/BIWEEKLY)
  "recurringEndDate": "2026-06-15T00:00:00Z",     ← required if isRecurring=true
  "contactEmail": "host@example.com",             ← optional
  "contactPhone": "+91-9999999999",               ← optional
  "websiteUrl": "https://...",                    ← optional
  "isCommentEnabled": true                        ← optional, default: true
}

→ Event is created in DRAFT status (not visible to public yet)
→ Creator is auto-added as "host" organizer
→ If isRecurring=true: system creates up to 52 child events automatically
→ Returns the full event object

Step 2: Add Ticket Types (optional, for paid events)

POST /api/ecosystem/event/:eventId/ticket-type
Auth: Required (host only)

Body:
{

  "description": "Limited offer", ← optional
  "price": 299,                   ← optional, default: 0 (free)
  "currency": "INR",              ← optional, default: INR

  "saleStartDate": "2026-02-01T00:00:00Z",  ← optional
  "saleEndDate": "2026-03-10T23:59:00Z",    ← optional
  "isActive": true,               ← optional, default: true
  "order": 0                      ← optional (display order)
}

→ Create multiple ticket types: "Free", "Early Bird", "VIP", etc.
→ Returns the created ticket type object

Step 3: Add Speakers (optional)

Body:

  "title": "CTO",                ← optional
  "company": "TechCorp",         ← optional
  "bio": "Expert in AI...",      ← optional
  "photo": "https://s3.../...",  ← optional (S3 URL)
  "topic": "Future of AI",       ← optional

}

Step 4: Add Sponsors (optional)

Step 5: Add Timeline/Agenda (optional)
─────────────────────────────────────────────────────────────────────────────

Auth: Required (host only)

Step 6: Add Media (optional)
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/media
Auth: Required (host only)

Body:
{
  "url": "https://s3.../image.jpg",  ← required (S3 URL)
  "mediaType": "IMAGE",              ← optional, default: IMAGE
  "caption": "Venue photo",          ← optional
  "order": 0                         ← optional
}

Step 7: Add Organizers (optional)
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/organizer
Auth: Required (host only)

Body:
{
  "userId": "user-uuid",   ← required (existing platform user)
  "role": "co_host"        ← optional, default: "organizer"
}

→ Role options: host, co_host, moderator, speaker, organizer

Step 8: Publish Event
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/publish
Auth: Required (host only)

No body needed.

→ Validates required fields: title, description, startDate, endDate, eventType
→ Changes status from DRAFT → PUBLISHED
→ Event becomes visible to everyone
→ If recurring: all child events are published too

Step 9: Update Event (anytime before completion)
─────────────────────────────────────────────────────────────────────────────
PATCH /api/ecosystem/event/:eventId
Auth: Required (host only)

Body: Any fields from Step 1 (partial update)
→ Cannot update COMPLETED or CANCELLED events
→ Cannot set capacity below current registration count

Step 10: Update Recurring Events
─────────────────────────────────────────────────────────────────────────────
PATCH /api/ecosystem/event/:eventId/recurring?mode=this_only
Auth: Required (host only)

Query param "mode":
  this_only   → updates just this one event (default)
  all_future  → updates this + all future child events
  all         → updates parent + all children

Body: Same as Step 9

Step 11: Duplicate Event
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/duplicate
Auth: Required (host only)

No body needed.
→ Creates a DRAFT copy with "(Copy)" appended to title
→ Does NOT copy registrations, team, tickets

Step 12: Update Ticket Type
─────────────────────────────────────────────────────────────────────────────
PATCH /api/ecosystem/event/ticket-type/:ticketTypeId
Auth: Required (host only)

Body: Any ticket fields (partial update)
→ Cannot set quantity below already sold count

Step 13: Delete Ticket Type
─────────────────────────────────────────────────────────────────────────────
DELETE /api/ecosystem/event/ticket-type/:ticketTypeId
Auth: Required (host only)

→ Cannot delete if tickets have been sold (quantitySold > 0)

Step 14: Update Speaker / Sponsor / Organizer / Timeline / Media
─────────────────────────────────────────────────────────────────────────────
PATCH /api/ecosystem/event/:eventId/speaker/:speakerId
PATCH /api/ecosystem/event/:eventId/sponsor/:sponsorId
PATCH /api/ecosystem/event/:eventId/organizer/:organizerId
PATCH /api/ecosystem/event/:eventId/timeline/:timelineId

Body: Same fields as create (partial update)

Step 15: Remove Speaker / Sponsor / Organizer / Timeline / Media
─────────────────────────────────────────────────────────────────────────────
DELETE /api/ecosystem/event/:eventId/speaker/:speakerId
DELETE /api/ecosystem/event/:eventId/sponsor/:sponsorId
DELETE /api/ecosystem/event/:eventId/organizer/:organizerId
DELETE /api/ecosystem/event/:eventId/timeline/:timelineId
DELETE /api/ecosystem/event/:eventId/media/:mediaId

→ Cannot remove the "host" organizer

Step 16: Send Update to Attendees
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/update
Auth: Required (host only)

Body:
{
  "title": "Venue Change Notice",       ← required
  "content": "The venue has changed...", ← required
  "notifyRegistrants": true              ← optional, default: true
}

Step 17: Cancel Event
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/cancel
Auth: Required (host only)

Body:
{
  "reason": "Unable to arrange venue"  ← optional
}

→ Cannot cancel COMPLETED or already CANCELLED events
→ All active registrations are cancelled too
→ If recurring: all child events + their registrations cancelled

Step 18: Complete Event
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/complete
Auth: Required (host only)

No body needed.
→ Only PUBLISHED events can be completed
→ Also happens automatically when endDate passes (every 15 min cron)

Step 19: Delete Event (Soft Delete)
─────────────────────────────────────────────────────────────────────────────
DELETE /api/ecosystem/event/:eventId
Auth: Required (host only)

→ Only DRAFT or CANCELLED events can be deleted
→ Soft delete (isArchived=true), not permanently removed
→ If recurring: all children archived too

6. FLOW 2: DISCOVER & BROWSE EVENTS
================================================================================

These are the listing/discovery APIs. Most work without login.

6.1 Browse All Events
─────────────────────────────────────────────────────────────────────────────
GET /api/ecosystem/event/all
Auth: Optional

Query Parameters (all optional):
  page=1                    ← pagination (default: 1)
  limit=10                  ← items per page (default: 10)
  search=tech               ← search in title, description
  sortBy=startDate          ← default: startDate
  order=asc                 ← asc or desc (default: asc)
  eventType=WORKSHOP        ← filter by event type
  format=VIRTUAL            ← filter by format
  city=Bangalore            ← filter by city
  country=India             ← filter by country
  isPaid=true               ← filter free vs paid
  isFeatured=true           ← filter featured events
  parentOnly=true           ← hide recurring children
  tag=tech                  ← filter by tag
  dateFrom=2026-03-01       ← events starting after this date
  dateTo=2026-03-31         ← events starting before this date
  upcoming=true             ← only future events (startDate >= now)

→ Only shows PUBLISHED events
→ If logged in: includes isBookmarked per event
→ Paginated response

6.2 View Event Details (by slug)
─────────────────────────────────────────────────────────────────────────────
GET /api/ecosystem/event/:slug
Auth: Optional

→ Full event detail: author, page, tickets, organizers, speakers, sponsors,

7. FLOW 3: FREE EVENT REGISTRATION
================================================================================

Step 1: Register
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/register
Auth: Required

Body:
{
  "ticketTypeId": "ticket-uuid"    ← optional (for ticketed events, even free ones)
}

What happens:
  → Event must be PUBLISHED
  → Event must not have started
  → Registration deadline must not have passed
  → Cannot register twice (unless previous was cancelled)

  Registration status depends on event settings:
  ┌─────────────────────────────────────────────────────────┐
  │ Condition                    → Status                    │
  ├─────────────────────────────────────────────────────────┤
  │ Event full + waitlist ON    → WAITLISTED                │
  │ Event full + no waitlist    → Error 400 "Full capacity" │
  │ requiresApproval = true     → REGISTERED (pending)      │
  │ autoConfirm = true          → CONFIRMED                 │
  │ default                     → REGISTERED                │
  └─────────────────────────────────────────────────────────┘

Response (free event):
{
  "data": {
    "registration": { "id": "...", "status": "CONFIRMED", ... },
    "requiresPayment": false
  }
}

Step 2: Check Status (anytime)
─────────────────────────────────────────────────────────────────────────────
GET /api/ecosystem/event/:eventId/registration-status
Auth: Required

Response:
{
  "data": {
    "id": "reg-uuid",
    "status": "CONFIRMED",
    "ticketName": "Free",
    "ticketPrice": 0,
    "payment": null
  }
}

→ Returns null if not registered

8. FLOW 4: PAID EVENT REGISTRATION (RAZORPAY)
================================================================================

Step 1: Register (with ticket)
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/register
Auth: Required

Body:
{
  "ticketTypeId": "vip-ticket-uuid"
}

→ Creates registration with status REGISTERED (pending payment)
→ Creates Razorpay order
→ Creates payment record with status PENDING

Response (paid event):
{
  "data": {
    "registration": { "id": "...", "status": "REGISTERED", ... },
    "requiresPayment": true,
    "payment": {
      "id": "payment-uuid",
      "amount": 999,
      "currency": "INR",
      "razorpayOrderId": "order_xxxxx",
      "status": "PENDING"
    },
    "razorpayOrder": {
      "id": "order_xxxxx",
      "amount": 99900,         ← amount in paise (999 * 100)
      "currency": "INR"
    }
  }
}

Step 2: Open Razorpay Checkout (Frontend)
─────────────────────────────────────────────────────────────────────────────
Use the razorpayOrder from Step 1 response:

const options = {
  key: RAZORPAY_KEY_ID,           // from env
  amount: response.razorpayOrder.amount,
  currency: "INR",
  order_id: response.razorpayOrder.id,
  handler: function(result) {
    // result contains: razorpay_payment_id, razorpay_order_id, razorpay_signature
    // Call Step 3 with these values
  }
};
const rzp = new Razorpay(options);
rzp.open();

Step 3: Verify Payment
─────────────────────────────────────────────────────────────────────────────
POST /api/ecosystem/event/:eventId/verify-payment
Auth: Required

Body:
{
  "razorpayPaymentId": "pay_xxxxx",       ← from Razorpay callback
  "razorpayOrderId": "order_xxxxx",       ← from Razorpay callback
  "razorpaySignature": "signature_xxxxx"  ← from Razorpay callback
}

→ Verifies payment signature
→ Payment status → COMPLETED
→ Registration status:
    - requiresApproval = true  → stays REGISTERED (host must approve)
    - requiresApproval = false → CONFIRMED (done!)

Step 4: Check Registration Status
─────────────────────────────────────────────────────────────────────────────
GET /api/ecosystem/event/:eventId/registration-status
Auth: Required

→ Confirm status is CONFIRMED (or REGISTERED if awaiting host approval)

STALE PAYMENT NOTE:
  If user doesn't complete payment within 30 minutes, the system automatically:
  → Marks payment as FAILED
  → Cancels the registration
  → Restores ticket quantity

================================================================================

These APIs are only accessible to the event HOST.

9.1 View All Registrations
─────────────────────────────────────────────────────────────────────────────
GET /api/ecosystem/event/:eventId/registrations
Auth: Required (host only)

Query: page, limit, search, status, ticketName

→ search works on: firstName, lastName, email

Response data per registration:
{
  "id": "reg-uuid",
  "status": "REGISTERED",
  "user": { "id", "firstName", "lastName", "email", "profilePhoto" },

  "payment": { "status": "COMPLETED", "amount": 999, "paidAt": "..." }
}

9.2 Registration Summary
─────────────────────────────────────────────────────────────────────────────
GET /api/ecosystem/event/:eventId/registrations/summary
Auth: Required (host only)

Response:
{
  "data": {
    "REGISTERED": 5,
    "CONFIRMED": 42,
    "WAITLISTED": 3,
    "CANCELLED": 2,
    "ATTENDED": 38,
    "NO_SHOW": 4
  }
}

11. FLOW 7: SOCKET.IO REAL-TIME INTERACTION
================================================================================

Used DURING a live event for chat, polls, and Q&A.
Must be used ALONGSIDE the Jitsi embed (Step 3 in FLOW 6).

CONNECTION SETUP:
─────────────────────────────────────────────────────────────────────────────
import { io } from "socket.io-client";

const socket = io("https://YOUR_BACKEND_URL/events", {
  auth: { token: accessToken }     ← same JWT used for REST APIs
});

socket.on("connect", () => {
  // Connected! Now join the event room
  socket.emit("join-live", { eventId: "event-uuid" });
});

socket.on("connect_error", (err) => {
  // Authentication failed
  console.error(err.message);
});

EVENTS YOU SEND (emit):
─────────────────────────────────────────────────────────────────────────────

join-live
  Send: { "eventId": "event-uuid" }
  Receive: "joined-live" { "success": true, "role": "PARTICIPANT" }

leave-live
  Send: { "eventId": "event-uuid" }
  (no direct response, viewer-count broadcast fires)

live-chat
  Send: { "eventId": "event-uuid", "message": "Hello everyone!" }
  (message is NOT saved to database - ephemeral only)

create-poll (HOST ONLY)
  Send: { "eventId": "event-uuid", "question": "Best framework?", "options": ["React", "Vue", "Angular"] }
  Broadcast: "new-poll"

vote-poll
  Send: { "pollId": "poll-uuid", "selectedOption": 0 }
  (one vote per user per poll)
  Broadcast: "poll-update"

end-poll (HOST ONLY)
  Send: { "pollId": "poll-uuid" }
  Broadcast: "poll-ended"

qa-submit
  Send: { "eventId": "event-uuid", "question": "How does this work?" }
  (question IS saved to database)
  Broadcast: "new-question"

qa-upvote
  Send: { "questionId": "q-uuid", "eventId": "event-uuid" }
  Broadcast: "question-upvoted"

qa-answer (HOST ONLY)
  Send: { "questionId": "q-uuid", "answer": "It works like...", "eventId": "event-uuid" }
  Broadcast: "question-answered"

EVENTS YOU LISTEN TO (on):
─────────────────────────────────────────────────────────────────────────────

joined-live
  Data: { "success": true, "role": "HOST" | "PARTICIPANT" | ... }

viewer-count
  Data: { "eventId": "...", "count": 42 }
  (fires on every join/leave)

live-chat-message
  Data: {
    "userId": "user-uuid",

    "message": "Hello!",

  }

new-poll
  Data: {
    "pollId": "poll-uuid",
    "question": "Best framework?",
    "options": ["React", "Vue", "Angular"],
    "totalVotes": 0
  }

poll-update
  Data: {
    "pollId": "poll-uuid",
    "totalVotes": 25,
    "results": [
      { "option": 0, "count": 12 },
      { "option": 1, "count": 8 },
      { "option": 2, "count": 5 }
    ]
  }

poll-ended
  Data: (same as poll-update, final results)

new-question
  Data: {
    "id": "q-uuid",
    "question": "How does this work?",
    "userId": "user-uuid",
    "userName": "John Doe",

  Data: { "id": "q-uuid", "answer": "It works like...", "answeredBy": "Jane Host" }

  Data: { "id": "q-uuid", "upvoteCount": 5 }

error
  Data: { "message": "Only the host can create polls" }

14. AUTOMATED BACKGROUND TASKS (GOOD TO KNOW)
================================================================================

These run automatically. No API call needed. Frontend should just handle
the state changes they produce.

Every 5 minutes:  WAITLIST PROMOTION
  → If someone cancels and capacity opens, waitlisted people get promoted

Every 10 minutes: STALE PAYMENT CLEANUP
  → Payments not completed within 30 min → FAILED
  → Registration cancelled, ticket restored

Every 15 minutes: AUTO-COMPLETE EVENTS
  → PUBLISHED events where endDate has passed → COMPLETED

Every hour:       EVENT REMINDERS
  → 24-hour reminders flagged
  → 1-hour reminders flagged
