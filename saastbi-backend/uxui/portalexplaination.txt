1. PLATFORM OVERVIEW
================================================================================

EcoSync is a comprehensive startup ecosystem platform that connects startups,
mentors, investors, VC firms, freelancers, incubators, and professionals in a
unified digital environment.

The platform operates as TWO distinct products:

PRODUCT A: ECOSYSTEM PLATFORM (ecosync.co.in)
  - A single, integrated platform where all user types (startups, mentors,
    investors, professionals, students, freelancers, VC partners) coexist
  - Think of it as "LinkedIn meets AngelList" for the startup ecosystem
  - Features: Social networking, job board, events, communities, mentorship
    marketplace, networking/discovery, and messaging

PRODUCT B: INCUBATION PORTAL (SaaS - e.g., iit.admin.ecosync.co.in)
  - A multi-tenant SaaS product for incubators/accelerators
  - Each incubator gets their own white-labeled portal (custom subdomain)
  - Features: Program management, startup pipeline, evaluation panels,
    office/facility management, funding disbursement, task management,
    mentor management, and announcements
  - Connected to the ecosystem via shared user accounts and data bridges

Both products share a common backend and database, but have completely separate
frontend interfaces, authentication flows, and user experiences.

2. PORTAL ARCHITECTURE
================================================================================

2.1 API STRUCTURE
-----------------
The backend organizes endpoints by portal context:

  /api/ecosystem/*          -> Ecosystem Platform APIs
  /api/incubation/*         -> Incubation admin onboarding/auth APIs
  /api/incubation-portal/*  -> Incubation Portal feature APIs (requires tenantKey header)
  /api/startup-portal/*     -> Startup-side view of incubation features
  /api/mentor/*             -> Mentor's own management portal
  /api/super-admin/*        -> EcoSync internal admin

2.2 MULTI-TENANCY MODEL (Incubation Portal)
--------------------------------------------
Each incubator is a "Tenant" with:
  - A unique tenantKey (used as subdomain: {tenantKey}.admin.ecosync.co.in)
  - Optional custom domain support
  - Status: ACTIVE, INACTIVE, SUSPENDED
  - Its own subscription plan, users, roles, programs, offices, and data
  - Tenant data is isolated; one incubator cannot see another's data
  - Redis caching for tenant lookups (24-hour TTL)

Authentication for the incubation portal requires BOTH:
  1. A valid tenantKey in the request header (identifies which incubator)
  2. A valid JWT token (identifies which user within that incubator)

2.3 ECOSYSTEM PLATFORM ARCHITECTURE
-------------------------------------
The ecosystem platform uses standard JWT authentication:
  - Users sign up with email/password or OAuth (Google, Facebook, Apple)
  - Users can have MULTIPLE roles simultaneously (e.g., a Founder who is also
    a Mentor and an Investor)
  - Role-based access controls determine what features and data are visible
  - Optional authentication on discovery/browsing endpoints (better experience
    when logged in, but browsable without account)

3. USER ROLES & PROFILES
================================================================================

3.1 ECOSYSTEM USER ROLES (PortalUserRole enum)
------------------------------------------------
Users can hold one or more of these roles:

  STUDENT
    - Looking for internships and career opportunities
    - Has: internship interests, career goals, preferred industries
    - Can: Browse jobs, apply for internships, join communities, attend events

  PROFESSIONAL
    - Working professional seeking opportunities
    - Has: open-to-opportunities flag, preferred roles/locations, salary
      expectations, notice period, remote preference
    - Can: Browse/apply for jobs, network, post content, join communities

  FREELANCER
    - Independent contractor/consultant
    - Has: hourly rate, availability status, portfolio, categories, ratings,
      total projects, total earnings, payout accounts
    - Can: Receive gig requests, manage projects, track milestones, earn and
      withdraw money, manage contracts

  FOUNDER
    - Startup founder/co-founder
    - Has: startup page(s), team members, startup metrics, pitch decks,
      funding rounds, investor updates
    - Can: Create startup pages, apply to incubation programs, book mentors,
      post jobs, manage startup team, request office space

  MENTOR
    - Experienced professional offering mentorship
    - Has: mentor profile (expertise, industries, session types, pricing,
      availability calendar, reviews/ratings, packages)
    - Can: Accept sessions, manage mentees, create packages, track earnings,
      associate with incubators, conduct video sessions (Jitsi)

  INVESTOR
    - Angel investor or individual investor
    - Has: investor profile (investment stages, sectors, check sizes, deal flow,
      portfolio, syndicate memberships)
    - Can: Manage deal pipeline (sourced -> screening -> due diligence ->
      term sheet -> invested/passed), organize syndicates, track portfolio

  VC_PARTNER
    - Venture capital firm partner/analyst
    - Has: VC profile (firm role, focus sectors/stages, deals sourced/closed,
      IC voting rights, term approval authority)
    - Can: Manage VC firm deal flow, fund management, portfolio monitoring,
      LP relations, team management (with granular permissions)

  INCUBATION_PERSON
    - Staff member at an incubator/accelerator
    - Bridge role connecting ecosystem profile to incubation portal user

3.2 INCUBATION PORTAL ROLES
-----------------------------
Within each tenant (incubator), users have these characteristics:

  Admin (isAdmin: true)
    - Full access to all tenant features
    - Can manage users, roles, programs, office spaces, facilities
    - Can create and manage announcements, view all startups
    - Can manage mentor associations and funding

  Panel Member (isPanelMember: true)
    - Assigned to evaluate startup applications for specific programs
    - Can submit evaluations with scoring against defined criteria
    - Can view application details for programs they're assigned to

  Regular User (with custom Role)
    - Access controlled by custom roles with module-based permissions
    - Each role has specific permissions (e.g., "can manage programs",
      "can view startups", "can book facilities")

Permission System:
  Module -> Permission -> RolePermission (role-based) + UserPermission (user-based)
  - Permissions are defined per module (e.g., "Programs" module has
    "create_program", "view_program", "manage_applications" permissions)
  - Roles aggregate permissions
  - Individual users can also have direct permission overrides

3.3 STARTUP MEMBER ROLES
--------------------------
Within a startup team:
  OWNER   - Full control, can manage all startup settings
  ADMIN   - Administrative access, can manage team and applications
  MEMBER  - Standard access, can view and contribute
  OTHER   - Custom role with specific permissions

3.4 COMMUNITY ROLES
---------------------
Within each community:
  OWNER     - Community creator, full control
  ADMIN     - Can manage settings, channels, rules, members
  MODERATOR - Can moderate posts, manage reports, ban/mute members
  MEMBER    - Can post, comment, engage with content

3.5 SUPER ADMIN ROLES
-----------------------
EcoSync internal staff:
  ADMIN               - Full platform control
  MARKETING           - Marketing operations
  MANAGER             - Operations management
  RELATIONSHIP_MANAGER - Tenant relationship management

4.1 AUTHENTICATION & ONBOARDING
---------------------------------
Registration Flow:
  1. User enters email, password, first name, last name
  2. OTP sent to email for verification
  3. User verifies OTP
  4. User selects a username (availability check provided)
  5. User selects their PRIMARY role (Student, Professional, Founder, etc.)
  6. User can add additional roles later
  7. Profile completion stages tracked (profileCurrentStage: 0-N)

Login Options:
  - Email/password login
  - Google OAuth
  - Facebook OAuth
  - Apple Sign-In
  - Social accounts can be linked/unlinked after registration

Password Recovery:
  - Forgot password -> OTP to email -> Verify OTP -> Reset password

Session Management:
  - JWT access tokens + refresh tokens
  - Cookie-based token storage supported
  - Token refresh endpoint for seamless session extension

Basic Information:

Education:
  - Institution (linked to organization Pages)

Experience:

Certifications:
  - Issuing organization (linked to Pages)

Projects:

Role-Specific Profiles:

Profile Analytics:

4.3 SOCIAL FEED & POSTS
-------------------------
Post Types:
  - TEXT: Plain text posts
  - IMAGE: Posts with one or more images
  - VIDEO: Posts with video content
  - DOCUMENT: Posts with attached documents
  - POLL: Interactive polls with multiple options
  - ARTICLE: Long-form articles
  - LINK: Link shares with preview metadata

Post Features:
  - Media attachments (via S3 pre-signed URL uploads)
  - Hashtags (auto-tracked, trending calculation)
  - User mentions (@username)
  - Post visibility: PUBLIC, CONNECTIONS_ONLY, PRIVATE
  - Pin posts to profile
  - Edit posts (tracked with isEdited flag)
  - Archive posts

Engagement:
  - Like/unlike posts
  - Comment (threaded, with parent/child replies)
  - Like/unlike comments
  - Share/repost with optional comment
  - Bookmark/save posts
  - Poll voting (single or multiple choice)
  - Stories (ephemeral content with view tracking)

Feed System:
  The feed uses a sophisticated scoring algorithm with multiple feed types:

  1. Personalized Feed - Algorithm-driven based on:
     - User interests and preferences
     - Connection/following activity
     - Engagement history (dwell time, scroll depth, link clicks)
     - Content freshness and trending score
     - Muted keywords filtering

  2. Following Feed - Posts from people you follow
  3. Trending Feed - Currently popular content
  4. Discover Feed - Content outside your network
  5. Topic Feed - Posts by specific hashtag/topic
  6. Connections Feed - Posts from connections only

  Feed Preferences (user-configurable):
  - Show connections only / following only
  - Interest topics
  - Muted keywords
  - Preferred content types

  Feed Actions:
  - Hide post (with reason: not interested, seen too often, spam, etc.)
  - Report post
  - Mute user

  Engagement Tracking (per post view):
  - Dwell time (seconds spent viewing)
  - Scroll depth (% of post viewed)
  - Link clicks
  - Media views

4.4 SOCIAL NETWORKING & CONNECTIONS
-------------------------------------
Relationship Types:

  Follow (asymmetric):
  - Any user can follow another
  - Follower sees followee's public posts in feed
  - No approval required

  Connection (symmetric):
  - Send connection request with optional message
  - Recipient can accept or reject
  - Once connected, both see each other's content
  - Can withdraw pending requests

  Block:
  - Block user with optional reason
  - Blocked user cannot see your content or interact
  - Unblock available

  Mute:
  - Mute user's content from your feed
  - User is not notified

Features:
  - Mutual followers/following/connections discovery
  - Batch relationship checking (up to 100 users at once)
  - Privacy settings control who can send connection requests
  - Message permissions: EVERYONE, CONNECTIONS, NO_ONE

4.5 NETWORKING & DISCOVERY
----------------------------
The networking module provides intelligent discovery across the ecosystem:

Discovery Categories:
  1. People - Discover professionals, founders, mentors, investors
  2. Startups - Discover startup companies
  3. Incubators - Discover incubation centers and their programs
  4. Pages - Discover organization pages (companies, institutions, VCs)

Features:
  - Global search across all entity types
  - Co-founder matching with preference system:
    - Set preferences: skills needed, commitment level, location, equity split
    - Algorithm matches compatible co-founder seekers
  - Network connections (your existing network)
  - AI-powered suggestions (people you may want to connect with)
  - Save/unsave profiles for later
  - Dismiss profiles from suggestions

Scoring Algorithm:
  The discovery system uses weighted scoring to rank results based on:
  - Profile completeness
  - Mutual connections
  - Shared interests/skills
  - Activity level
  - Location proximity
  - Role compatibility

-----------------------

  1. Check if tenant name/key is available

  - Managers oversee their assigned programs

Application Workflow:
  1. Startup discovers program (via ecosystem or direct link)
  2. Startup fills application (answers scheme questions, uploads documents)
  3. Application status: NEW
  4. Incubation admin reviews application
  5. Can request changes (ChangeRequest):
     - Types: GENERAL, FILE, QUESTION
     - Startup responds with changes (ChangeResponse)
     - Status: PENDING -> RESOLVED/REJECTED
  6. Can request documents (DocumentRequest):
     - Startup uploads documents (DocumentResponse)
  7. Application moves through stages:
     NEW -> CHANGES_REQUESTED -> CHANGES_RECEIVED -> REVIEWED ->
     UNDER_EVALUATION -> EVALUATED -> DOCS_REQUESTED -> DOCS_RECEIVED ->
     VERIFIED -> ONBOARDED (or REJECTED at any stage)
  8. Panel evaluation:
     - Panel members assigned to program
     - Each panel member scores application against evaluation criteria
     - Total score calculated with weightages
     - Evaluation summary generated
  9. On approval: Startup onboarded to incubation portal

Application History:
  - Every status change is logged with timestamp, changed-by user, and comment
  - Complete audit trail

5.5 PANEL EVALUATION SYSTEM
-----------------------------
A structured evaluation workflow for startup applications:

Setup:
  - Invite panel members (from incubation users with isPanelMember flag)
  - Assign panel members to specific programs
  - Define evaluation questions with:
    - Question text
    - Weightage (importance multiplier)
    - Multiple choice options, each with a score
    - Scoring reference/rubric

Evaluation Flow:
  1. Panel member sees assigned applications
  2. Reviews application details
  3. Scores each evaluation question
  4. Adds optional comments per question
  5. Submits evaluation (status: DRAFT -> SUBMITTED)
  6. Each application can be evaluated by multiple panel members
  7. Evaluation summary aggregates all evaluators' scores

Output:
  - Per-evaluator breakdown
  - Average scores per question
  - Total weighted score
  - Ranking capability

14. PAYMENT & SUBSCRIPTION ARCHITECTURE
================================================================================

All payments are processed through Razorpay (Indian payment gateway):

14.1 PAYMENT SCENARIOS
------------------------

  Incubation Subscriptions:
  - Recurring subscription (Razorpay subscription API)
  - Plan-based pricing
  - Webhook for payment events

  Mentor Sessions:
  - One-time Razorpay order
  - Order creation -> Payment -> Signature verification
  - Platform fee deduction
  - Mentor payout tracking

  Mentor Packages:
  - One-time payment for session bundle
  - Same Razorpay flow

  Event Tickets:
  - One-time payment for ticket purchase
  - Refund support for cancellations

  Office Space:
  - Booking payments (rent, security deposit)
  - Razorpay order flow
  - Refund handling

  Freelancer Payments:
  - Milestone-based payments
  - Platform fee deduction
  - Payout to freelancer bank/UPI

14.2 PAYMENT STATUSES
-----------------------
  PENDING -> CAPTURED -> SETTLED (or FAILED/REFUNDED)

14.3 PAYOUT SYSTEM
---------------------
  For mentors and freelancers:
  - Track earnings per transaction
  - Platform fee, GST, TDS deductions
  - Withdrawal requests
  - Bank transfer or UPI payout via Razorpay
  - Payout account verification

17. DESIGN CONSIDERATIONS & RECOMMENDATIONS
================================================================================

17.1 NAVIGATION COMPLEXITY
-----------------------------
The platform has TWO completely separate products that need distinct navigation:

  Ecosystem Platform:
  - Primary navigation: Feed, Network, Jobs, Events, Communities, Messages
  - Secondary: Profile, Mentor Marketplace, Notifications, Settings
  - Role-based sections: Mentor Dashboard, Investor Dashboard, etc.
  - Page management (for organization pages)

  Incubation Portal:
  - Primary navigation: Dashboard, Programs, Startups, Tasks
  - Secondary: Office Spaces, Facilities, Mentors, Funding, Announcements
  - Admin: Users, Roles, Settings, Subscription
  - Each tenant has its own URL (subdomain or custom domain)

17.2 ROLE-BASED UI ADAPTATION
---------------------------------
The ecosystem platform must adapt based on user roles:
  - A FOUNDER sees startup management, program applications, mentor booking
  - A MENTOR sees session management, availability, earnings
  - An INVESTOR sees deal flow, portfolio, syndicates
  - A STUDENT sees internships, career tools
  - Users can have MULTIPLE roles -- the UI should allow switching context

  Recommendation: Use a role switcher or combined dashboard that surfaces
  relevant features per active role.

17.3 MULTI-TENANCY UI CONSIDERATIONS
---------------------------------------
  - Each incubator portal should feel like its own product
  - Tenant logo, branding colors should be customizable
  - The tenantKey in subdomain identifies the organization
  - Login flow requires tenant identification first
  - Data is completely isolated between tenants

17.4 KEY UX PATTERNS TO CONSIDER
-----------------------------------

  Application Pipeline (Kanban):
  - Startup applications move through stages
  - Visual pipeline view would be highly effective
  - Similar to hiring pipeline in job management

  Calendar Views:
  - Mentor availability
  - Session scheduling
  - Event calendar
  - Facility booking calendar
  - Office space availability

  Dashboard Patterns:
  - Incubation admin dashboard (key metrics, recent activity)
  - Mentor dashboard (upcoming sessions, earnings)
  - Startup dashboard (application status, tasks)
  - Investor dashboard (deal pipeline, portfolio)
  - Community analytics dashboard

  Chat/Messaging:
  - Conversation list with unread badges
  - Message threading
  - Context-aware chat (mentor session, mentorship)
  - Real-time indicators

  Notification Center:
  - Bell icon with unread count
  - Category-based grouping
  - Action buttons for quick response
  - Mark as read/archive

  Profile Completeness:
  - Progress indicators for profile completion
  - Guided setup flows for new users
  - Stage-based onboarding

17.5 DATA-HEAVY VIEWS
------------------------
Several views involve large data tables that need thoughtful UX:
  - Program applications list (with status filters, search)
  - Job applications pipeline
  - Event registrations
  - Community members list
  - Mentor sessions history
  - Earnings/payout history
  - Funding disbursements
  - Office bookings
  - Task boards (kanban or list view)

  Recommendation: Use consistent table patterns with:
  - Search and filter bars
  - Status badges with color coding
  - Pagination
  - Sort options
  - Bulk actions where applicable
  - Export functionality

17.6 MOBILE CONSIDERATIONS
-----------------------------
  - The ecosystem platform (feed, networking, messaging, events) is heavily
    mobile-first in user behavior
  - The incubation portal is more desktop-oriented (admin operations)
  - Mentor booking and session joining should work well on mobile
  - Push notifications for time-sensitive actions (session reminders, etc.)

17.7 PAYMENT UX
-----------------
  - Razorpay checkout integration (modal/redirect)
  - Payment status indicators
  - Invoice download
  - Subscription management (upgrade/downgrade/cancel)
  - Earnings dashboard with charts

17.8 REAL-TIME FEATURES
--------------------------
  - Chat messages (instant delivery)
  - Notifications (badge updates)
  - Live event streaming
  - Online status indicators

17.9 FORM COMPLEXITY
-----------------------
Several complex forms exist:
  - Program creation (many fields, file uploads)
  - Event creation (multi-step with tickets, speakers, timeline)
  - Job posting (with screening questions)
  - Startup application (dynamic questions from scheme)
  - Mentor profile setup
  - Investor deal flow entry

  Recommendation: Use multi-step forms with save-as-draft capability
  for complex creation flows.

17.10 CONTENT MODERATION UI
------------------------------
  - Moderation queue for communities
  - Report review workflow
  - Ban/mute member actions
  - Content approval flow (posts in moderated communities)
  - Activity logs for audit

17.11 ANALYTICS & REPORTING
------------------------------
Several analytics views are needed:
  - Profile insights (views, engagement)
  - Page analytics (followers, engagement)
  - Event analytics (registrations, attendance)
  - Community analytics (growth, engagement, retention)
  - Hiring dashboard (application funnel)
  - Mentor analytics (sessions, earnings, ratings)
  - Funding analytics (sources, allocations, disbursements)
  - Facility usage statistics
  - Office space occupancy

17.12 SEARCH & DISCOVERY UX
-------------------------------
  - Global search across entities (people, startups, pages, jobs, events)
  - Filter panels with multiple criteria
  - Sort options
  - Saved searches (job alerts)
  - Recommendation cards
  - Trending/featured sections
  - Infinite scroll or pagination

17.13 ACCESSIBILITY & INTERNATIONALIZATION
---------------------------------------------
  - Language support (default English, configurable)
  - Currency support (INR primary, USD for international)
  - Timezone handling (default Asia/Kolkata)
  - Date format considerations
  - Color contrast for status badges and alerts
  - Keyboard navigation for power users
